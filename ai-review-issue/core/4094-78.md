## 이슈 요약

native AVX/NEON CPU raster 최적화를 `wasm_simd128.h` 기반 WebAssembly SIMD로 포팅하고 주요 browser에서 성능을 측정하자는 요청이다. 기존 #2211과 사실상 같은 범위다.

## main 코드 조사

- 최상위 `meson.build`의 `simd` 분기는 x86 AVX와 ARM NEON만 설정한다. Emscripten/`__wasm_simd128__` 분기가 없다.
- 구현은 `src/renderer/cpu_engine/tvgSwRasterAvx.h`와 `tvgSwRasterNeon.h`, 선택은 `tvgSwRaster.cpp`의 compile-time macro로 이뤄진다.
- scalar fallback은 존재하지만 wasm SIMD binary/runtime 선택 정책은 없다.

```text
packed ABGR pixels
scalar: [p0] [p1] [p2] [p3] ...
wasm128: <p0 | p1 | p2 | p3>  + scalar tail
AVX256:  <p0 | p1 | ... | p7>  (직역 불가)
```

```cpp
// current compile-time dispatch의 형태
#if defined(THORVG_AVX_VECTOR_SUPPORT)
    return avxRasterTranslucentRect(...);
#elif defined(THORVG_NEON_VECTOR_SUPPORT)
    return neonRasterTranslucentRect(...);
#else
    return cRasterTranslucentRect(...);
#endif
```

## 원인 가설

**추론:** 픽셀 blend/fill 루프는 128-bit lane으로 옮길 수 있으나 AVX 256-bit 구현을 기계적으로 번역할 수는 없다. browser가 이미 자동 vectorize하는 부분과 hand-written intrinsic의 코드 크기 차이를 측정해야 한다.

## 수정 방향과 실현 가능성

1. #2211과 중복 정리 후 하나의 benchmark/acceptance 기준을 사용한다.
2. hot function 한 개를 `-msimd128`로 prototype하고 scalar bit-exact 비교를 만든다.
3. SIMD/비SIMD binary 제공 정책과 runtime 호환성을 결정한다.
4. 첫 prototype은 32-bit translucent rectangle처럼 입출력 계약이 단순한 hot loop 하나로 제한하고 랜덤 buffer differential test를 만든다.
5. `-msimd128` 단일 artifact와 SIMD/non-SIMD 이중 배포의 size/cache 비용을 함께 보고한다.

## 위험/검증

SIMD 미지원 runtime의 module load 실패, unaligned/tail access, Chrome/Firefox/Safari 편차가 있다. FPS와 wasm size를 함께 보고해야 한다.
