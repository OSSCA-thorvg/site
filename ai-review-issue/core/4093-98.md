## 이슈 요약

GL/WG의 4× MSAA 비용을 줄이고 얇은 선/곡선 품질을 높이기 위해 analytical edge coverage 방식으로 교체하자는 연구·최적화 제안이다.

## main 코드 조사

- GL `tvgGlRenderTarget.cpp`는 color/depth-stencil multisample renderbuffer를 sample 4로 만든다.
- WG `tvgWgPipelines.cpp`의 기본 multisample state count도 4이고 `WgRenderTarget`이 4-sample attachment를 만든다.
- 현재 tessellator는 fill/stroke triangle mesh를 만들고 stencil/cover pipeline으로 그린다. analytical edge distance나 per-vertex coverage 데이터는 없다.
- CPU RLE coverage는 구조가 달라 GPU에 그대로 포팅할 수 없다.

```text
현재: path -> triangle tessellation -> 4 samples/pixel -> resolve

후보: path -> edge/fringe 또는 curve params -> fragment distance/coverage -> 1 sample
                                            └-> stencil/clip/blend와 결합 필요
```

| 항목 | SW | GL | WG |
|---|---|---|---|
| AA 표현 | scanline/RLE coverage | 4× multisample color+depth/stencil renderbuffer | pipeline `.count=4` + multisampled target |
| geometry | CPU outline/scanline | tessellated fill/stroke | `WgBWTessellator`/stroker |
| analytical 전환 | reference가 될 수 있음 | vertex/shader/target 모두 변경 | WGSL/pipeline/target 모두 변경 |

## 원인 가설

**추론:** 새 AA는 tessellator가 edge/fringe geometry 또는 curve 파라미터를 shader에 전달하고, stencil/clip/blend와 함께 coverage를 계산해야 한다. 단순 sample count 변경이 아니다.

## 수정 방향과 실현 가능성

1. MSAA GPU time/memory와 품질 baseline을 장치별로 측정한다.
2. line/convex path 한 종류에 analytical fringe prototype을 만든다.
3. cubic, self-intersection, clip/mask까지 확장 가능성과 binary/shader cost를 평가한다.
4. 기존 MSAA와 feature/experiment flag로 병렬 비교하고 GPU별 실패 corpus가 없는 상태에서 기본값을 바꾸지 않는다.
5. 품질 metric과 목표 GPU time/memory threshold를 RFC acceptance로 고정한다.

## 위험/검증

모바일 tile GPU, WebGL precision, crack/overlap, painter order와 모든 blend mode를 검증해야 한다. 성능 이득을 보장할 수 없는 renderer 연구 과제다.
