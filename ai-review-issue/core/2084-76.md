## 이슈 요약

화면 크기보다 극단적으로 큰 stroke 폭을 지정해도 보이는 영역에 비례한 비용으로 처리하자는 성능·안정성 이슈다. 현재 코드는 RLE bbox는 clip하지만 거대한 stroke outline을 만든 뒤에야 clip하므로, “전체 화면 크기의 RLE buffer를 늘린다”와 “clip 전 기하 계산이 비싸거나 overflow한다”를 구분해야 한다.

## main 코드 조사

### 확인된 증거

- `SwShapeTask::validStrokeWidth()`는 transform scale을 곱한 폭을 계산하지만 viewport 기준 upper bound는 두지 않는다.
- `shapeGenStrokeRle()`는 `strokeParseOutline()`과 `strokeExportOutline()`로 확장 윤곽을 만든 뒤 `utilExport()`하고, 그 다음에야 `utilBBox()`로 `clipBox`와 교차한다.
- `strokeReset()`은 `strokeWidth()/2`를 그대로 join/cap 좌표 생성에 쓴다.
- `utilExport()`는 좌표에 64를 곱해 `int32_t` fixed point로 cast한다. 매우 큰 값에서 범위 문제가 생길 수 있는 지점이다.
- RLE `reqSize`는 이미 **교차 후 `renderBox`**의 최대 변 길이로 계산된다. 그러므로 현재 코드에서 cell pool이 무조건 1e8 폭으로 잡힌다고 단정하면 안 된다.

```cpp
// shapeGenStrokeRle(): clip은 outline을 모두 만든 뒤 적용된다.
strokeParseOutline(shape.stroke, *outline, mpool, tid);
outline = strokeExportOutline(shape.stroke, mpool, tid);
utilExport(outline, transform, bbox);
if (!utilBBox(bbox, clipBox, renderBox, false)) return false;

// utilExport(): 거대한 float 좌표를 26.6 fixed int32로 바꾼다.
outline->out.push({int32_t(t.x * 64.0f), int32_t(t.y * 64.0f)});
```

### 아직 확인되지 않은 부분

- 폭 1e7/1e8에서 실제 wall time, peak RSS, 호출별 profile과 UBSan 결과를 아직 측정하지 않았다.
- 병목이 outline point 수, miter/join 수치, fixed conversion, RLE band retry 중 어디인지는 코드 읽기만으로 확정되지 않는다.

## 원인 가설

| 후보 | 코드 근거 | 상태 |
|---|---|---|
| clip 전 거대 outline 좌표 계산 | `strokeParseOutline()`이 `utilBBox()`보다 먼저다. | 강한 가설 |
| int32 fixed 범위 초과 | `t.x * 64`를 `int32_t`로 cast한다. | 강한 위험, 재현 필요 |
| 전체 폭만큼 RLE cell pool 할당 | `rleRender()`는 이미 clipped `renderBox`를 받는다. | 현재 코드로는 약한 가설 |
| 특정 miter/dash에서 point 폭증 | style별 분기와 curve split이 존재한다. | profile 필요 |

## 수정 방향과 실현 가능성

1. 이슈 예제를 독립 benchmark로 만들고 폭 `10^2...10^8`, Bevel/Round/Miter, transform 조합의 시간·RSS를 기록한다.
2. `strokeParseOutline`, `utilExport`, `rleRender` 구간을 따로 계측하고 ASan/UBSan을 실행한다.
3. 화면을 완전히 덮는 **불투명 단색·비대시 stroke**처럼 증명 가능한 경우만 rect coverage fast path로 줄이는 안을 검토한다.
4. 일반 경로는 viewport를 고려한 safe coordinate domain 또는 double intermediate를 검토하되, geometry를 단순 clamp하지 않는다.
5. 최적화 전후 output을 다양한 cap/join/dash/gradient/clip에서 pixel diff하고 성능 수치를 보고한다.

```text
path -> stroke outline(현재 큰 좌표) -> float×64/int32 -> bbox∩viewport -> RLE
             ^ 후보 병목/overflow                    ^ allocation은 여기서 제한됨
```

## 위험과 검증

- “화면을 채운다”는 판정은 transparent/gradient/dash stroke에는 적용할 수 없다.
- miter는 폭 이외에 miter limit과 각도에 따라 더 멀리 돌출될 수 있다.
- 성능 개선은 작은 정상 stroke가 느려지지 않는지도 함께 측정해야 한다.
