## 이슈 요약

첨부 Lottie의 GPU anti-aliasing 경계에서 색/alpha가 잘못 합성되는 현상이다. premultiplied alpha target에서는 부분 coverage 픽셀도 `rgb <= alpha`를 만족하고, coverage가 RGB와 alpha에 같은 횟수로 반영되어야 한다.

#4197은 모든 engine/blend mode의 공통 정합성 이슈이고, 이 문서는 #4113의 GPU 재현을 좁혀 진단하는 작업으로 본다.

## 현재 main 코드 조사

### 확인된 사실

- [`WgPipelines`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/gpu_engine/wg/tvgWgPipelines.cpp)는 4× MSAA를 사용하고 normal color/alpha blend를 모두 `One / OneMinusSrcAlpha`로 설정한다. 이는 fragment output이 premultiplied라는 계약이다.
- [`WgShaderSrc`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/gpu_engine/wg/tvgWgShaderSrc.cpp)의 solid/gradient 경로는 대체로 `vec4(Sc.rgb * Sc.a * So, Sc.a * So)`를 반환한다.
- [`GlShaderSrc`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/gpu_engine/gl/tvgGlShaderSrc.cpp)의 solid color도 `vec4(color.rgb * color.a, color.a)`를 출력하고, GL sync blend도 `GL_ONE, GL_ONE_MINUS_SRC_ALPHA`를 사용한다.
- [`GlRenderTarget`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/gpu_engine/gl/tvgGlRenderTarget.cpp) 역시 color/depth renderbuffer를 4 samples로 만들고 resolve한다.
- GL에는 `strokeRenderWidth < MIN_GL_STROKE_WIDTH`일 때 alpha를 폭 비율만큼 낮추는 별도 thin-stroke 보정이 [`GlSolidBatch`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/gpu_engine/gl/tvgGlSolidBatch.cpp)와 [`GlRenderer`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/gpu_engine/gl/tvgGlRenderer.cpp)에 있다.
- WGPU에는 solid/gradient/image/scene 및 custom blend가 서로 다른 shader/pipeline로 분기한다. 한 shader만 수정해 전체 GPU 경로가 맞는다고 보장할 수 없다.

정상적인 premultiplied source-over 기준은 다음과 같다.

```text
effectiveAlpha = sourceAlpha × objectOpacity × coverage
premultRgb     = straightRgb × effectiveAlpha
outRgb         = premultRgb + dstRgb × (1 - effectiveAlpha)
outAlpha       = effectiveAlpha + dstAlpha × (1 - effectiveAlpha)
```

backend 비교:

| 단계 | GL | WG | 점검 포인트 |
|---|---|---|---|
| AA | 4× MSAA renderbuffer | 4× multisample state | sample coverage가 resolve에서 한 번만 적용되는가 |
| color alpha | shader에서 RGB×A | shader에서 RGB×A×opacity | opacity가 RGB/A에 대칭인가 |
| thin stroke | 폭을 alpha로 추가 축소 | 동일한 명시 경로는 확인되지 않음 | MSAA coverage와 이중 감쇠되는가 |
| normal blend | One / OneMinusSrcAlpha | One / OneMinusSrcAlpha | input이 실제 premultiplied인가 |

### 아직 가설인 부분

- 실제 첨부 현상이 thin-stroke alpha 보정 때문이라는 것은 아직 가설이다. fill, gradient, mask 또는 custom blend 장면일 수도 있다.
- MSAA resolve가 coverage를 잘못 계산한다기보다 resolve 전 fragment RGB/A가 이미 불일치할 가능성도 있다.
- sRGB/linear 변환이나 최종 blit의 straight/premultiplied target 차이가 색 번짐을 만든 것인지 GPU capture 전에는 확정할 수 없다.
- GL과 WG가 동일하게 실패하는지, 한 backend만 실패하는지 로컬 증거가 없다.

## 수정 방향과 실현 가능성

실현 가능성은 **중간**이다. 첨부를 확보하거나 동등한 최소 scene을 만든 뒤에는 좁혀갈 수 있지만, 수식 확인 없이 alpha 곱을 추가/제거하면 double-premultiplication이 된다.

1. solid fill, solid 0.5 px stroke, gradient stroke, group opacity를 각각 분리한 최소 scene을 만든다.
2. transparent와 opaque 배경에서 GL/WG fragment output, resolved offscreen texture, final target의 RGBA를 캡처한다.
3. 각 경로에서 `source alpha`, `object opacity`, `geometric/thin-stroke coverage`, `MSAA coverage`를 표로 기록한다.
4. 최종 fragment output 직전 한 지점에서만 color alpha/opacity premultiplication이 이뤄지도록 한다. geometric sample coverage는 MSAA에 맡길지 alpha 보정에 넣을지 경로별로 명시한다.
5. normal blend를 먼저 고정하고 custom blend는 #4197의 cross-engine conformance 작업으로 분리한다.

GPU capture에서 먼저 확인할 불변식은 간단하다.

```cpp
// 0..1 normalized fragment output
assert(rgb.r <= alpha + epsilon);
assert(rgb.g <= alpha + epsilon);
assert(rgb.b <= alpha + epsilon);
```

## 위험과 검증 계획

- stroke width 0.25/0.5/1.0/2.0 px, opacity 0/0.5/1.0, alpha 0/0.5/1.0 조합을 검사한다.
- MSAA resolve 전 sample과 resolve 후 pixel을 분리해 본다.
- solid/linear/radial/image, mask/matte, offscreen scene, normal/custom blend를 확인한다.
- transparent black과 opaque white/black 배경에서 reference pixel을 비교한다.
- SW 결과를 시각 기준으로 쓰되, SW와 GPU의 AA 방식이 달라 exact pixel equality 대신 불변식과 tolerance를 사용한다.
- GL desktop/GLES와 WG validation layer에서 같은 fixture를 실행한다.
