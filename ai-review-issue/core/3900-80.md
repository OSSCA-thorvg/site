# #3900 Upgrade GL dependency to v4

- Link: https://github.com/thorvg/thorvg/issues/3900
- 난이도: 80/100
- 실현 가능성: 낮음
- 초심자 추천: 비추천
- 관련 영역: OpenGL profile/version, GLES/WebGL fallback, shader/function loader
- 배울 수 있는 것: capability detection, optional symbols, GLSL variants, portability

## 이슈 요약

desktop 최소 GL을 4.x로 올리자는 전략 제안이지만 필요한 GL4 feature가 특정되지 않았다. GLES3/WebGL2를 유지하려면 단순 version bump가 아니라 capability별 fast path/fallback 설계가 필요하다.

## main 코드 조사

**확인된 증거**

- README와 `tvgGl.h`는 desktop GL 3.3, GLES 3.0을 최소로 둔다.
- `glInit()`는 runtime major/minor를 검사한다.
- shader prefix는 desktop `#version 330 core`, GLES/WebGL `#version 300 es`다.
- Emscripten은 GLES/WebGL 경로를 사용하고 GL functions는 자체 table에서 load된다.

```cpp
#if defined(THORVG_GL_TARGET_GLES)
    #define TVG_REQUIRE_GL_MAJOR_VER 3
    #define TVG_REQUIRE_GL_MINOR_VER 0
#else
    #define TVG_REQUIRE_GL_MAJOR_VER 3
    #define TVG_REQUIRE_GL_MINOR_VER 3
#endif
```

| target | 현재 baseline | GL4 공통화 가능성 |
|---|---|---|
| desktop GL | 3.3 core | optional fast path 가능 |
| GLES | 3.0 | GL4 API 직접 사용 불가 |
| WebGL2 | GLES 계열 | browser 허용 API만 가능 |

## 원인 가설과 확인 방법

- **확정:** 현재 하나의 portable baseline과 compile-time GL/GLES variant가 있다.
- **가설:** 필요한 기능은 GL4 core가 아니라 GL3 extension/capability로도 제공 가능하다.
- **확인 방법:** 먼저 원하는 feature의 required function/extension/shader construct를 표로 만든다.

## 수정 방향 계획

1. GL4가 필요한 concrete feature와 measured benefit을 정의한다.
2. minimum version bump와 optional capability fast path를 지원 범위/size로 비교한다.
3. 새 symbols는 optional load로 만들고 GL3.3/GLES/WebGL fallback을 유지한다.
4. shader source/version도 capability별로 최소 분기하고 platform matrix를 구축한다.

## 실현 가능성 판단

요구 feature가 없어 현재는 **낮음**이다. version 숫자만 올리는 patch는 완료 기준과 이득이 불명확하다.

## 위험/검증

- unsupported symbol이 전체 GL init을 실패시키지 않아야 한다.
- Windows/Linux/macOS, GLES와 Emscripten에서 compile/init/reference render를 확인한다.
- shader variants와 binary-size 증가 및 실제 GPU benefit을 측정한다.
