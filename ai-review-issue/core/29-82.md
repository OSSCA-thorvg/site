- 배울 수 있는 것: premultiplied alpha, RLE span, AVX/SSE intrinsic, 정렬·tail 처리, 런타임 CPU dispatch

## 이슈 요약

픽셀 단위 scalar 루프가 남아 있는 소프트웨어 렌더러를 AVX로 최적화하자는 장기 이슈다. 이슈 본문도 함수별 작은 패치를 권한다. 현재 main에는 fill과 반투명 rect/RLE용 AVX 코드가 있지만, source pixel 배열을 합성하는 두 함수와 gradient 경로 등은 scalar로 남아 있다. 따라서 이슈 전체는 아직 유효하지만, 성공 조건을 “모든 raster 함수의 AVX화”로 잡으면 끝이 없는 성능 프로젝트가 된다.

## main 코드 조사

### 확인된 사실

- [`tvgSwRaster.cpp`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwRaster.cpp)의 `rasterTranslucentPixel32()`와 source-array 버전 `rasterPixel32()`는 명시적으로 scalar C 함수를 호출하며 SIMD TODO가 남아 있다.
- [`tvgSwRasterAvx.h`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwRasterAvx.h)에는 grayscale/solid pixel fill과 translucent rect/RLE가 있다. fill은 256-bit store를 쓰지만 alpha blend 본체는 128-bit `__m128i` 네 픽셀 단위다.
- 최상위 [`meson.build`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/meson.build)는 `simd=true`와 x86 host를 보고 `THORVG_AVX_VECTOR_SUPPORT`를 정의한다. [`src/meson.build`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/meson.build)는 이 경우 `-mavx`를 ThorVG library 전체의 compile flag에 넣는다.
- 코드 검색상 CPUID/XGETBV 기반 런타임 dispatch는 없다. 즉 현재 선택은 runtime이 아니라 build-time 선택이다.

현재 scalar 진입점은 다음처럼 분명하다.

```cpp
void rasterTranslucentPixel32(uint32_t* dst, uint32_t* src,
                              uint32_t len, uint8_t opacity)
{
    //TODO: Support SIMD accelerations
    cRasterTranslucentPixels(dst, src, len, opacity);
}
```

```text
-Dsimd=false ──> scalar C

-Dsimd=true + x86 ──> library 전체 -mavx
                         └─> compile-time AVX 경로
                             (runtime CPU fallback 없음)
```

### 아직 가설인 부분

- **가설 A:** `rasterTranslucentPixel32()`가 post-effect와 image 합성에서 호출되므로 좋은 첫 후보일 수 있다. 그러나 profile 없이 가장 큰 병목이라고 확정할 수 없다.
- **가설 B:** 긴 span에는 SIMD 이득이 있지만 짧은 span에서는 broadcast/alignment 비용 때문에 scalar보다 느릴 수 있다.
- **가설 C:** 단순히 AVX 함수를 추가하는 것보다 baseline object와 AVX object를 분리한 runtime dispatch가 호환성 측면에서 선행 과제일 수 있다. 이 설계는 현재 코드에 구현되어 있지 않다.

## 수정 방향과 실현 가능성

1. `rasterTranslucentPixel32()` 하나만 대상으로 호출 횟수, 평균 `len`, 소비 시간을 profile한다.
2. scalar 함수를 reference로 두고 alpha `0, 1, 127, 254, 255`, 길이 `0..vector-width+1`, 비정렬 주소를 자동 비교한다.
3. SIMD body와 scalar tail을 독립 구현하되 [`tvgSwCommon.h`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwCommon.h)의 `ALPHA_BLEND()` 반올림을 bit-exact하게 유지한다.
4. 별도 object/translation unit과 CPUID dispatch가 필요한지 먼저 결정한다. library 전체 `-mavx`를 그대로 두고 “runtime fallback”이라고 주장해서는 안 된다.
5. 동일 compiler/옵션/asset으로 scalar 대비 CPU time과 binary-size delta를 기록한다.

**판정:** 한 함수의 reference test와 prototype은 실현 가능하다. 그러나 이슈 제목 수준의 전체 최적화와 안전한 runtime dispatch까지 포함하면 저사양 x86 장비와 성능 분석 경험이 필요해 실현 가능성은 낮다.

## 위험과 검증 체크리스트

- AVX 미지원 x86 및 OS가 AVX register state를 활성화하지 않은 환경
- ABGR/ARGB, premultiplied/straight 변환 경계
- alpha와 coverage 경계값, 길이 0 및 scalar tail
- 4/16/32-byte 비정렬 주소와 sanitizer
- 긴 rect, 짧은 RLE span, post-effect 등 서로 다른 workload
- 성능 수치와 binary-size 수치를 포함한 claim 검증
