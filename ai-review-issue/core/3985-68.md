## 이슈 요약

focal point가 중심에서 크게 벗어난 radial gradient에서 CPU는 부드럽지만 GL/WG는 경계나 banding이 더 두드러지는 품질 이슈다.

## main 코드 조사

- `src/renderer/tvgFill.h::RadialGradientImpl::correct()`는 focal point가 end circle 밖/경계에 있으면 안쪽으로 clamp하고 start radius도 보정한다.
- CPU `src/renderer/cpu_engine/tvgSwFill.cpp::_prepareRadial()/fillRadial()`은 보정값과 특이점용 분기를 사용한다.
- GL은 `tvgGlShaderSrc.cpp::compute_radial_t()`에서 여러 radial/strip/focal-on-circle 분기를 가진다.
- WG `tvgWgShaderSrc.cpp`는 이차방정식 `(-b + sqrt(d))/(2*a)`를 직접 계산한다. `WgShaderTypeGradient::update()`도 focal correction을 호출하지만 `a≈0` 특이점 분기는 shader에 없다.

```wgsl
// 현재 WG 일반식: a가 0에 가까우면 분모가 불안정하다.
let d = b * b - 4.0 * a * c;
if (d >= 0.0) { t = min(1.0, (-b + sqrt(d)) / (2.0 * a)); }
```

| 단계 | SW | GL | WG |
|---|---|---|---|
| focal 보정 | `RadialGradientImpl::correct()` 사용 | 동일 | 동일 |
| `t` 계산 | CPU 고정소수점 준비/증분 | radial/strip/일반 특수 분기 | 일반 이차식 중심 |
| 핵심 위험 | reference 자체의 정확성 | 분기별 경계 연속성 | `a≈0`, discriminant와 GPU precision |

## 원인 가설

**추론:** 첨부 SVG는 focal이 circle 밖에 있어 correction 후 경계에 가까운 수치가 된다. WG의 `2*a`가 매우 작아지는 경우와 GL의 별도 근사식이 CPU보다 정밀도에 민감한 것이 후보이다. 두 GPU가 완전히 같은 원인인지는 pixel/probe 값으로 확인해야 한다.

## 수정 방향과 실현 가능성

1. focal 위치를 중심→경계→외부로 이동한 pixel golden test를 만든다.
2. CPU/GL/WG에 전달되는 corrected 좌표와 주요 `a,b,c,d,t` 값을 비교한다.
3. 특이점 판정을 CPU에서 공통 uniform으로 계산하거나 안정된 공식을 두 shader에 맞춘다.

첫 PR은 fixture와 shader 입력/중간값 계측까지만 분리할 수 있다. 공식 변경은 GPU별 오차가 동일한 원인임을 확인한 뒤 진행하는 편이 안전하다.

## 위험/검증

spread mode, focal radius, transform, low precision 모바일 GPU를 검사해야 한다. CPU 결과를 reference로 삼되 성능과 shader size 변화도 측정한다. 초심자는 fixture/수치 조사부터 조건부로 적합하다.
