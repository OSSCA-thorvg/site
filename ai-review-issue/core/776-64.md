# #776 — sw_engine: scaling very small SVGs

- Link: https://github.com/thorvg/thorvg/issues/776
- 난이도: 64/100
- 실현 가능성: 중간
- 초심자 추천: 조건부 (재현 test와 최초 손실 지점 추적까지)
- 분석 기준: `main` working tree `f989b27892ba`
- 관련 영역: SVG stroke, CPU fixed-point rasterization, anti-aliasing
- 배울 수 있는 것: SVG viewBox/scale, stroke outline, 26.6 fixed-point 좌표, RLE coverage

## 이슈 요약

좌표 범위가 2×2이고 `stroke-width=0.05/0.1`인 SVG를 크게 표시했을 때 CPU 결과가 브라우저와 다르다는 재현 이슈다. 작은 입력이 본문에 그대로 있어 테스트 시작점은 좋지만, current main에서 동일 증상이 남았는지 실행 증거는 이 로컬 조사에 포함되어 있지 않다. 코드는 stroke geometry의 작은-segment 판정과 transform 후 1/64 pixel 양자화를 모두 사용하므로 최초 손실 단계를 구분해야 한다.

## main 코드 조사

### 확인된 사실

- [`tvgSvgBuilder.cpp`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/loaders/svg/tvgSvgBuilder.cpp)의 `_applyStroke()`는 파싱된 width/cap/join/miter/dash를 `Shape`에 전달한다.
- [`tvgSwStroke.cpp`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwStroke.cpp)의 `_tiny()`는 양축이 `2/64`보다 작은 vector를 tiny로 간주하고 여러 stroke subdivision/중복점 제거 경로에서 사용한다.
- stroke outline은 source 좌표에서 만든 뒤 [`utilExport()`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwUtil.cpp)에서 transform되고 `int32_t(t.x * 64.0f)`로 잘려 26.6 fixed-point가 된다.
- [`tvgSwRle.cpp`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwRle.cpp)가 변환된 outline을 coverage span으로 바꾼다.
- [`tvgSwRenderer.cpp`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwRenderer.cpp)의 `validStrokeWidth()`는 transform scale을 반영해 AA 정책에 쓰지만, 이 값이 `_tiny()` 임계값을 바꾸지는 않는다.

두 정밀도 관문은 서로 다른 단계에 있다.

```cpp
// stroke geometry 단계: source coordinate 기준
constexpr float EPSILON = 2.0f / 64.0f;
return fabsf(pt.x) < EPSILON && fabsf(pt.y) < EPSILON;

// raster export 단계: transform 후 device coordinate 기준
auto t = *pt * transform;
outline->out.push({int32_t(t.x * 64.0f), int32_t(t.y * 64.0f)});
```

### backend별 진단 분기

| 관찰 | SW | GL | WG | 더 강한 후보 |
|---|---|---|---|---|
| SW만 틀림 | 26.6 export/RLE/CPU AA 사용 | GPU geometry | GPU geometry | CPU quantization·RLE |
| 세 backend 모두 틀림 | 공통 SVG model/path 사용 | 공통 model/path 사용 | 공통 model/path 사용 | parser, stroke geometry, transform |
| scale에 따라 단계적으로 변함 | fixed-point 가능성 큼 | tessellation tolerance 가능 | tessellation tolerance 가능 | 최초 분기 scale을 계측 |

### 아직 가설인 부분

- **가설 A:** source 좌표 기준 `_tiny()`가 확대 전에 유효 segment를 제거할 수 있다. 실제 재현 path가 어느 `_tiny()` 호출에서 탈락하는지 trace하지 않았으므로 미확정이다.
- **가설 B:** transform 후 26.6 truncation이 coverage 비대칭을 만들 수 있다. 확대 scale에서 좌표 간격이 충분히 커진다면 이 가설은 약해질 수 있다.
- **가설 C:** GL/WG에서도 같다면 SVG path parsing 또는 공통 stroke semantics 문제일 수 있다. backend image가 없는 상태에서 SW 문제라고 단정하지 않는다.

## 수정 방향과 실현 가능성

1. 이슈 본문의 SVG를 resource로 고정하고 scale과 stroke width를 sweep한 CPU golden test를 만든다.
2. 동일 build에서 GL/WG 이미지를 얻어 “공통 단계 vs CPU 전용 단계”를 먼저 나눈다.
3. SVG build 직후 path/stroke width, `strokeParseOutline()` 후 점 수, `utilExport()` 후 fixed-point 좌표를 순서대로 기록한다.
4. 점이나 segment가 처음 사라지는 함수 하나만 수정한다. 전역 epsilon 변경부터 시도하지 않는다.
5. dash/cap/join/degenerate path와 기존 stroke tests를 함께 실행한다.

**판정:** test 작성과 trace는 초심자도 실현 가능하다. 정확한 수정은 최초 손실 지점이 `_tiny`, export, RLE 중 어디인지에 따라 난도가 크게 달라진다.
