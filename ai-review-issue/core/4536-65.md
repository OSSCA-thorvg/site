## 이슈 요약

내장 PNG loader 결과의 색이 기대 결과와 눈에 띄게 다르다는 compliance 문제다.

## main 코드 조사

`src/loaders/png/tvgPngLoader.cpp::run()`은 lodepng 출력 형식을 `LCT_RGBA`로 요청한 뒤 surface를 `ColorSpace::ABGR8888S`로 표시한다. 실제 decode/color conversion은 vendored `tvgLodePng.cpp`에 있다. 비교 대상과 input asset의 ICC/gAMA/sRGB chunk 여부가 본문에는 없다.

| 경계 | 확인할 증거 |
|---|---|
| PNG → lodepng RGBA | reference decoder와 raw byte 비교 |
| RGBA → `ABGR8888S` surface | channel order와 straight alpha 확인 |
| Surface → renderer output | premultiply/upload 뒤 pixel 비교 |

## 원인 가설

가능성은 channel order보다 PNG color-management metadata(gAMA/sRGB/iCCP) 미적용 또는 decoder version 차이다. 이미지만으로 확정할 수 없다.

## 수정 방향 계획

1. 원본 PNG를 확보해 chunk와 pixel sample을 reference decoder와 비교한다.
2. decode 직후 RGBA bytes가 다른지, renderer upload 뒤에 다른지 경계를 나눈다.
3. metadata 미지원이라면 지원 범위와 binary-size 영향을 합의한다.
4. 작은 fixture와 exact pixel assertion을 추가한다.

## 위험/검증

재현 fixture가 확보되면 loader 학습 과제로 좋지만 color management 자체는 깊다. 모든 PNG에 보정하면 기존 결과가 바뀔 수 있어 test corpus가 필요하다.
