# #2884 gl_engine: binary shader support

- Link: https://github.com/thorvg/thorvg/issues/2884
- 난이도: 82/100
- 실현 가능성: 낮음
- 초심자 추천: 비추천
- 관련 영역: OpenGL program binary, 파일 캐시, driver 호환성, WebGL/embedded
- 배울 수 있는 것: shader compile/link, runtime capability probe, cache invalidation, backend 차이

## 이슈 요약

매 실행마다 GLSL을 컴파일·링크하는 비용을 줄이기 위해 program binary를 저장하고 다음 실행에서 재사용하자는 최적화 제안이다. 이전 문서의 “조사 보류”를 해제하고 현재 코드에서 필요한 일을 추적했지만, 저장된 maintainer 의견대로 성능 이득 증명이 acceptance 조건이라는 점은 남는다.

## main 코드 조사

**확인된 증거**

- `GlShader::compileShader()`는 실행 시 `glShaderSource()`와 `glCompileShader()`를 호출한다.
- `GlProgram` 생성자는 매번 vertex/fragment shader를 attach하고 `glLinkProgram()`한다.
- `tvgGl.h/.cpp`의 로드 함수 table에는 `glProgramBinary`, `glGetProgramBinary`, `glProgramParameteri`가 없다.
- renderer 초기화에는 기본 program만 15개 이상 생성되고, blend/effect program은 추가되거나 lazy 생성된다.
- 파일 I/O는 `THORVG_FILE_IO_SUPPORT`로 빠질 수 있고 Emscripten은 별도 WebGL 경로다. 현재 GL program cache abstraction은 없다.

```cpp
// src/renderer/gpu_engine/gl/tvgGlProgram.cpp
glAttachShader(progObj, shader.getVertexShader());
glAttachShader(progObj, shader.getFragmentShader());
glLinkProgram(progObj);
```

| backend | 현재 초기화 | 이 이슈의 직접 적용 가능성 |
|---|---|---|
| SW | shader 없음 | 해당 없음 |
| GL/GLES(native) | GLSL compile + program link | runtime capability가 있을 때만 가능 |
| GL/WebGL | GLSL 경로지만 브라우저 API 제약 | native program-binary cache와 같은 방식 불가 |
| WG | WGSL module과 다수 pipeline 생성 | 별도 WebGPU pipeline-cache 문제이며 이 이슈 범위 밖 |

## 원인 가설과 확인 방법

- **확정:** ThorVG GL 코드 자체는 source compile/link만 하며 binary load/save 경로가 없다.
- **가설:** startup에서 shader compile/link가 유의미한 병목이다. driver가 자체 cache를 제공하면 ThorVG cache의 순이득은 작을 수 있다.
- **확인 불가 범위:** 이 작업 환경에는 대표 GPU vendor/driver 행렬, cold driver-cache 제어, 애플리케이션 cache 경로 요구사항이 없다. 따라서 소스 조사는 가능하지만 “몇 ms 개선”과 cache hit 안정성은 확정할 수 없다.

## 수정 방향 계획

1. 코드 변경 전 `GlRenderer::initShaders()`와 lazy effect program의 compile/link 시간을 분리 계측한다.
2. native GL/GLES에서 program binary format 수와 필요한 entry point를 runtime probe한다. 미지원이면 기존 source 경로만 사용한다.
3. `shader source hash + GL/GLES variant + vendor/renderer/version + binary format + ThorVG build id`를 key로 삼는다.
4. binary load/link 검증 실패, 손상 파일, driver 변경 시 cache를 폐기하고 source compile로 자동 fallback한다.
5. cache write는 임시 파일 후 교체하고, file I/O off·WebGL에서는 코드를 compile out한다.

```text
capability 없음 ───────────────▶ source compile/link
capability 있음 ─▶ cache hit ─▶ binary load ─ 성공 ▶ 사용
                         └──── 실패 ───────────────▶ source fallback + 갱신
```

## 실현 가능성 판단

native GL 한 플랫폼의 prototype은 가능하다. 그러나 전체 이슈는 **낮음**이다. 저장된 이슈 내용상 제출 전 실효성 확인이 요구되고, 이 환경에서는 multi-driver benchmark를 수행할 수 없다. 초심자는 timer 계측과 capability probe를 별도 하위 작업으로 맡을 수 있지만 cache 정책 전체는 플랫폼 경험이 필요하다.

## 위험/검증

- cold/warm app start와 cold/warm driver cache를 구분해 측정한다.
- Mesa/NVIDIA/ANGLE 등 여러 구현, GL/GLES, cache 손상·권한 없음·driver update를 확인한다.
- WebGL, `-Dfile=false`, static/embedded 빌드에서 새 symbol이나 파일 의존성이 생기지 않아야 한다.
