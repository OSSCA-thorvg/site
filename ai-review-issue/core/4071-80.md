## main 코드 조사

- Lottie는 `tvgLottieParser.cpp`가 JSON model을 만들고 `tvgLottieBuilder.cpp`가 frame별 Scene/Shape/Text를 갱신한다.
- transform, masks, trim, text, modifiers, expressions 등 가능한 경로가 많다.
- 현재 loader는 JSON Lottie를 처리하며 `.lottie` container parser는 없다. 첨부는 JSON이므로 container 자체보다 콘텐츠 compliance 문제로 보인다.

```text
JSON bytes -> Parser -> Model/Property(frame) -> Builder/Paint tree -> SW | GL | WG
              값?       보간/시간?              누락/순서?           backend만?
```

현재 저장소와 `issues.json` 스냅샷에는 attachment URL만 있고 asset bytes는 없다. 이번 요청은 재크롤링을 금지했으므로 영상만 보고 특정 subsystem을 확정하지 않았다.

## 원인 가설

**불확실:** 영상과 링크만으로는 transform interpolation, text, mask, expression 중 어느 기능인지 특정할 수 없다. 외부 runtime도 같은 문제가 있는지에 따라 asset/spec 문제일 수도 있다.

## 수정 방향과 실현 가능성

1. reference runtime과 ThorVG의 첫 divergence frame을 찾는다.
2. 해당 frame의 visible layer를 하나씩 제거해 최소 JSON으로 줄인다.
3. 실패 property의 parser 값과 builder 결과를 비교한 뒤 수정 범위를 정한다.
4. SW/GL/WG가 모두 같게 틀리면 loader/builder를, 한 backend만 다르면 renderer를 우선 조사한다.
5. 최소 JSON과 최초 실패 frame을 golden test로 고정한 뒤에만 수정한다.

## 위험/검증

전체 asset만 golden test로 두면 원인과 회귀가 불명확하므로 최소 fixture가 필요하다.

## 추가 확인 필요

본문에 기대 동작과 실패 frame/property가 기술되지 않아 코드 시작점을 한 함수로 좁힐 수 없다. 다만 parser→model→builder→backend라는 진단 순서와 외부 자료 한계는 명확하므로 난이도를 80점으로 확정했다.
