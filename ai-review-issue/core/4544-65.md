- 배울 수 있는 것: callback ABI와 binary data contract

## 이슈 요약

nested base64 image를 resolver callback에서 받았지만 `src`가 decoded binary인데 길이가 없어 사용할 수 없다는 보고다. callback에 `src_size` 추가를 제안한다.

## main 코드 조사

`Tvg_Picture_Asset_Resolver`는 `thorvg_capi.h`에서 `(paint, const char* src, void* data)`이고, `tvgCapi.cpp`가 이를 C++ `Picture::resolver(std::function<bool(Paint*, const char*, void*)>)`로 adapter한다. 내부 `AssetResolver`도 size를 표현하지 않는다. 따라서 C API만이 아니라 C++ API, loader 호출부까지 contract 변경이 필요하다.

```c
/* 현재: path/URI와 binary payload를 구분할 길이/종류가 없다. */
typedef bool (*Tvg_Picture_Asset_Resolver)(
    Tvg_Paint paint, const char* src, void* data);
```

## 원인 가설

**추론:** `src`가 원래 path/URI null-terminated string이라는 전제로 설계됐지만 embedded decoded bytes까지 같은 parameter에 실으면서 길이 정보가 사라진 것으로 보인다.

## 수정 방향 계획

먼저 `src`가 path, data URI, decoded bytes 중 무엇인지 종류까지 명시하는 API를 설계한다. experimental API라도 C/C++ signature와 모든 호출부, 문서, CAPI test를 함께 갱신하고 기존 callback 호환 정책을 합의한다.

## 위험/검증

문제와 호출 경로는 명확해 학습 가치가 있지만 public ABI 변경이다. 단순 parameter 하나 추가로 끝내지 말고 binary/text 구분과 lifetime을 검증해야 하므로 mentor가 필요하다.
