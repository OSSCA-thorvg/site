# #3121 API: Introduce a Path class

- Link: https://github.com/thorvg/thorvg/issues/3121
- 난이도: 97/100
- 실현 가능성: 낮음
- 초심자 추천: 비추천
- 관련 영역: public Path/Shape API, ownership/COW, CAPI/ABI, renderer lifetime
- 배울 수 있는 것: 객체 수명, 공유/복제, abstraction cost, path boolean 기반

## 이슈 요약

Shape 내부 path를 독립 공개 `Path` 클래스로 분리해 재사용과 추후 boolean 연산을 제공하자는 API 제안이다. mutable Path 공유 의미, dirty propagation, renderer task lifetime과 C ABI를 새로 정해야 하므로 핵심 아키텍처 변경이다.

## main 코드 조사

**확인된 증거**

- 공개 `Shape`가 `moveTo/lineTo/cubicTo/close`, preset append와 raw path getter를 직접 제공한다.
- 내부 `ShapeImpl`은 `RenderShape`를 값으로 보유하고, 그 `RenderPath`는 자체 `Array<PathCommand>`/`Array<Point>` 두 개다.
- 현재 path ownership은 Shape와 함께 움직이며 duplicate는 별도 배열을 만든다. public `Path` type, shared ownership, COW와 boolean API는 없다.
- SW task는 `RenderShape*`를 비동기로 읽고 GPU도 같은 공통 path에서 geometry를 준비하므로 공유 Path mutation은 renderer lifetime과 충돌할 수 있다.

```cpp
// src/renderer/tvgRender.h
struct RenderPath {
    Array<PathCommand> cmds;
    Array<Point> pts;
    void moveTo(const Point& pt);
    void lineTo(const Point& pt);
    void cubicTo(const Point&, const Point&, const Point&);
};
```

| ownership 모델 | 장점 | 핵심 비용/위험 |
|---|---|---|
| attach 시 deep copy | 단순·독립적 | 재사용 메모리 이득이 거의 없음 |
| immutable shared Path | 수명/dirty가 단순 | 편집 시 새 객체 필요 |
| mutable + copy-on-write | 편집성과 공유 절충 | ref/COW/동시성 및 binary size 증가 |

## 원인 가설과 확인 방법

- **확정:** 현재 path는 Shape-owned value이며 독립 lifetime contract가 없다.
- **가설:** 반복 geometry에서 Path sharing이 메모리를 줄이고 API를 명료하게 한다.
- **반대 가설:** 객체/ref-count/COW overhead와 fragmentation이 작은 Shape에서는 이득을 상쇄한다.
- **확인 방법:** 세 ownership prototype을 반복 icon/큰 path/animated loader workload로 비교하고 memory, duplicate/update latency, binary size를 측정한다.

## 수정 방향 계획

1. boolean 연산을 첫 버전에서 분리하고 Path 생성/edit/attach/detach/duplicate/lifetime만 API proposal로 고정한다.
2. immutable 공유, COW, attach-copy를 benchmark와 사용례로 비교해 하나를 선택한다.
3. Shape dirty flag와 worker task가 참조하는 snapshot 규칙을 명세하고 concurrency test를 먼저 만든다.
4. C++ API와 opaque C handle/ref/unref contract, export/ABI 계획을 함께 리뷰한다.
5. primitives와 duplicate vertical slice 후 loaders/backends를 순차 이행한다.

## 실현 가능성 판단

단계적 구현은 가능하지만 현재 전체 이슈는 **낮음**이다. ownership과 public API 결정을 maintainer가 먼저 내려야 한다. 초심자는 benchmark fixture나 API usage prototype을 맡을 수 있으나 core 구현 담당으로는 적합하지 않다.

## 위험/검증

- shared Path 수정 시 다른 Shape로 변경이 새는지, detach/rel 후 UAF와 ref leak이 없는지 검사한다.
- update/render 동시성과 SW/GL/WG geometry cache invalidation을 stress-test한다.
- shared/static/WASM, CAPI link test와 binary/instance-size 변화를 측정한다.
