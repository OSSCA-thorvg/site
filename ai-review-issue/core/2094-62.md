## 이슈 요약

`moveTo(p); lineTo(p); close()` 또는 매우 짧은 닫힌 선분에 Round cap을 지정했을 때 점/원이 기대되지만 cap이 한쪽만 보이거나 사라지는 문제다. 구현 결함과 함께 “닫힌 contour에는 원래 cap이 있는가”라는 의미 결정이 필요하다.

## main 코드 조사

### 확인된 증거

- `_outlineClose()`는 시작점을 한 번 더 넣고 contour를 `closed=true`로 기록한다.
- `_lineTo()`는 zero-length를 감지하고 첫 점이며 cap이 Butt가 아니면 `_firstSubPath()`를 호출한다. 즉 “점도 cap으로 보인다”는 의도는 코드에 명시돼 있다.
- 그러나 `_endSubPath()`는 `closedSubPath`이면 join/close 경로를 타고, 열린 contour에만 `_addCap()`을 두 번 호출한다.
- `strokeParseOutline()`은 `outline.closed`를 그대로 `_beginSubPath()`에 넘긴다. 영 길이인지 별도 분류하지 않는다.
- test에는 cap API 값 검사는 있지만 이 이슈의 degenerate pixel regression은 검색되지 않았다.

```cpp
// zero-length를 점으로 취급하려는 코드
if (tvg::zero(delta)) {
    if (stroke.firstPt && stroke.cap != StrokeCap::Butt)
        _firstSubPath(stroke, 0.0f, 0.0f);
    return;
}

// 하지만 closed contour는 cap을 전혀 부르지 않는다.
if (stroke.closedSubPath) { /* join + close */ }
else { _addCap(...); /* ... */ _addCap(...); }
```

### 아직 확인되지 않은 부분

- 현재 main에서 이슈의 case 2~8을 렌더한 픽셀 결과를 다시 생성하지 않았다.
- SVG와 ThorVG Path API가 “closed zero-length path + round cap”에 같은 결과를 요구하는지는 이슈만으로 확정되지 않는다.

## 원인 가설

- **확인된 제어 흐름:** zero-length line은 `_firstSubPath()`까지 가지만 `closed=true` 때문에 cap 생성은 건너뛴다.
- **강한 가설:** 이 경로가 case 2의 사라짐/비정상 결과를 설명한다. 생성된 양쪽 border가 면적 없는 closed outline이 될 수 있다.
- **추가 가설:** 1px 닫힌 line은 시작점으로 되돌아오는 segment와 join 계산이 겹쳐 angle/rounding에 따라 비대칭이 된다.

```text
open point:   Move ─ zero Line ─ end(open)   -> cap + cap -> dot
closed point: Move ─ zero Line ─ Close       -> join/close -> cap 없음
```

## 수정 방향과 실현 가능성

1. 이슈의 8개 case를 각각 독립 shape로 만들고 Open/Closed × Butt/Round/Square 결과를 golden fixture로 추가한다.
2. `strokeParseOutline()`에서 유효 non-zero segment 수와 contour length를 기록해 degenerate contour를 명시적으로 분류한다.
3. 합의된 semantics가 점 렌더라면 zero-length closed contour에만 round 원/square 사각형 outline을 생성하는 제한된 fast path를 둔다.
4. 1px처럼 짧지만 non-zero인 closed contour는 별도 case로 유지해 일반 polygon의 cap 금지 원칙을 깨지 않는다.
5. duplicate point가 중간에 섞인 정상 polygon, dash, transform, 큰 폭을 회귀 검사한다.

## 위험과 검증

- 일반 closed polygon에 cap을 추가하는 수정은 명백한 회귀다.
- Butt cap의 zero-length path는 계속 보이지 않아야 하는지 테스트로 고정한다.
- SW 수정이므로 GL/WG와 결과가 다를 수 있다. 공통 semantics test를 만들되 backend별 raster tolerance를 둔다.
