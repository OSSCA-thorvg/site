## 이슈 요약

공식 Zig binding/package를 추가하려는 ecosystem 작업이다. raw C import MVP는 작지만 safe wrapper와 source build/registry/release까지 포함하면 범위가 커진다.

## main 코드 조사

**확인된 증거**

- language-neutral surface는 `thorvg_capi.h`와 `tvgCapi.cpp`다.
- Meson binding option은 현재 `capi`만 제공하며 versioned include directory에 header를 설치한다.
- C API는 opaque handles와 explicit create/destroy/add/remove 형태라 raw FFI의 기반이 있다.
- 저장소에는 Zig source, `build.zig` 또는 package metadata가 없다.

```mermaid
flowchart LR
    Z[Zig package] --> I[@cImport thorvg_capi.h]
    I --> C[libthorvg C ABI]
    Z --> O[optional safe ownership wrapper]
    Z --> B[system/static link helper]
```

```c
Tvg_Canvas tvg_swcanvas_create(Tvg_Engine_Option op);
Tvg_Result tvg_canvas_destroy(Tvg_Canvas canvas);
// Zig wrapper도 생성/해제 ownership을 보존해야 한다.
```

## 원인 가설과 확인 방법

- **확정:** C ABI와 installed header가 있어 raw binding은 가능한 구조다.
- **가설:** 가장 유지 가능한 MVP는 generated API 복사보다 `@cImport`+build/link helper다.
- **확인 방법:** installed ThorVG로 init→Shape→Canvas→draw→sync→destroy smoke program을 작성해 필요한 wrapper만 확인한다.

## 수정 방향 계획

1. MVP를 raw import, build helper, one smoke example로 제한해 maintainer 승인을 받는다.
2. system-installed/shared와 caller-provided static library 두 link mode를 prototype한다.
3. ownership과 Result/ColorSpace mapping만 얇은 wrapper로 제공할지 결정한다.
4. CI/release 지원 Zig version과 package ownership을 문서화한다.

## 실현 가능성 판단

MVP는 **중간**이며 Zig 경험이 있는 초심자에게 조건부 추천이다. full idiomatic wrapper/registry 운영까지 한 번에 맡기면 범위를 넘는다.

## 위험/검증

- Zig build API version, C enum/opaque pointer/callback ABI와 library discovery를 확인한다.
- Linux/macOS/Windows 및 static/shared smoke test를 분리한다.
- generated declarations를 복제한다면 C header drift 검사를 반드시 둔다.
