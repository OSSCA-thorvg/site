# #3133 — 그룹을 가로지르는 Lottie individual trim

- Link: https://github.com/thorvg/thorvg/issues/3133
- 난이도: 72/100
- 실현 가능성: 중간
- 초심자 추천: 조건부 (fixture 축소와 context trace부터)
- 분석 기준: `main` working tree `f989b27892ba`
- 관련 영역: Lottie group/model, RenderContext, trim-path length
- 배울 수 있는 것: modifier scope, simultaneous/individual trim, 계층 content traversal

## 이슈 요약

서로 다른 Lottie group의 shape들이 하나의 individual trim operator 범위에 있을 때 전체 path 집합의 누적 길이로 순차 trim되지 않고 simultaneous sample과 같은 결과가 나오는 compliance 이슈다. current renderer의 `RenderTrimPath`는 하나의 `RenderPath` 안에서는 두 모드를 구분하지만, Lottie builder가 group 경계에서 Shape/RenderContext를 분리하면 renderer 단계에서는 다른 group의 길이를 알 수 없다.

## main 코드 조사

### 확인된 사실

- [`LottieTrimpath`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/loaders/lottie/tvgLottieModel.h)은 `Simultaneous=1`, `Individual=2`를 보관한다.
- [`updateTrimpath()`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/loaders/lottie/tvgLottieBuilder.cpp)은 현재 context의 `propagator` 하나에 `trimpath(begin, end, type == Simultaneous)`를 설정한다. 전달받은 전체 `contexts`와 `parent`는 사용하지 않는다.
- `updateGroup()`은 각 group에 새 local `Inlist<RenderContext>`를 만들고 merge 가능하지 않으면 `propagator`를 duplicate한다.
- [`RenderTrimPath::trim()`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/tvgRender.cpp)은 `simultaneous=true`면 한 RenderPath의 subpath마다 같은 구간을 trim하고, false면 전체 RenderPath를 한 번에 trim한다.
- 이미 서로 다른 Shape로 분리된 group끼리의 길이를 `RenderTrimPath`가 합산하는 API는 없다.

```cpp
// current builder: 현재 context 하나에만 적용
ctx->propagator->trimpath(
    begin, end,
    trimpath->type == LottieTrimpath::Type::Simultaneous
);
```

```text
기대 individual: [group A path length][group B path length]
                    └──── 하나의 연속 구간으로 trim ────┘

현재 후보:        [Shape A에 0..t][Shape B에 0..t]
                    group마다 같은 구간 반복
```

### 아직 가설인 부분

- **강한 가설:** group별 propagator 분리 후 동일 individual 값이 각각 적용되어 simultaneous처럼 보인다.
- **가설 B:** `allowMerge`/`reqFragment` 상태에 따라 일부 group은 하나의 Shape로 합쳐져 증상이 asset 구조에 따라 달라질 수 있다.
- **가설 C:** builder prepass가 가장 자연스럽지만 animated path 길이를 매 frame 계산하면 성능 비용이 생길 수 있다.

## 수정 방향과 실현 가능성

1. attachment를 “nested group 2개 + path 1개씩 + individual trim 1개”로 줄여 local test resource로 만든다.
2. frame마다 만들어지는 RenderContext, Shape command count와 각 path 길이를 기록한다.
3. trim operator의 영향 범위에 있는 path를 먼저 수집하고 전체 길이 prefix sum을 만든다.
4. global `[begin,end]`을 각 group/Shape의 local interval로 변환하되 group transform은 길이 계산에 반영한다.
5. simultaneous 경로를 건드리지 않고 flat/nested, merge/fragment, repeater와 animated path를 검증한다.

**판정:** fixture 축소와 원인 확인은 초심자에게 가능하다. 최종 구현은 Lottie content traversal과 성능을 함께 고려해야 한다.
