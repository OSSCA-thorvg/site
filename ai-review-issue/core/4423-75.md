## 이슈 요약

CPU renderer에서 v1.0 이후 보인다는 raster artifact다. 로컬 자료만으로는 artifact가 image edge, mask seam, scale aliasing, alpha fringe 중 무엇인지 판독할 수 없다. 따라서 “CPU rasterizer 문제”는 영역 표기이지 원인이 확정됐다는 뜻이 아니다.

## main 코드 조사

Lottie image layer는 별도 raster 코드를 갖지 않고 `Picture`를 scene에 넣는다.

```cpp
auto picture = image->bitmap.picture;
if (!picture) return;
layer->scene->add(picture->refCnt() == 1 ? picture : picture->duplicate());
```

SW에서는 transform 모양에 따라 서로 다른 경로로 갈라진다.

```text
Picture / Lottie image
        |
        v
imagePrepare()
  +-- identity + translation ------> rasterDirectImage
  +-- axis-aligned scale ----------> rasterScaledImage
  |                                   +-- nearest
  |                                   +-- bilinear up-scale
  |                                   `-- mean-kernel down-scale
  `-- rotate/skew/perspective -----> rasterTexmapPolygon
                                      + clip/mask/composition
```

확인된 주요 분기:

- `imagePrepare()`는 off-diagonal matrix가 0이면 `scaled`, 아니면 texmap 경로를 택한다.
- `_scaleMethod()`는 filter와 `DOWN_SCALE_TOLERANCE`에 따라 nearest, bilinear, mean down-scaler를 고른다.
- RLE clip이 있으면 direct/scaled/texmap이 다시 별도 함수로 갈라진다.
- `SwRenderer::renderImage()`는 whole image와 RLE image를 각각 다른 함수로 보낸다.
- precomp는 `updatePrecomp()`에서 viewport clip을 적용하므로 artifact가 composition 경계일 가능성도 남는다.

### 백엔드 비교로 원인 범위를 줄이는 법

| 관찰 | 우선 조사 영역 |
|---|---|
| SW에서만 발생, GL/WG는 동일 | SW sampling/RLE/coverage/blend |
| 세 엔진 모두 같은 위치 | Lottie transform, image asset, precomp clip/mask |
| 축소에서만 발생 | `_interpDownScaler`, scale threshold, source alpha |
| 회전/기울임에서만 발생 | `tvgSwRasterTexmap.h` polygon sampling |
| layer 단독은 정상, 합성 후 발생 | mask/matte/premultiplied alpha composition |

## 원인 가설과 확인 방법

현재 직접 확정된 원인은 없다. 아래는 우선순위가 있는 조사 가설이다.

1. **scale sampler 전환:** 동일 transform을 threshold 전후로 움직여 artifact가 갑자기 생기는지 확인한다.
2. **clip/RLE seam:** mask와 precomp clip을 각각 제거하고 span 경계 좌표의 coverage를 덤프한다.
3. **premultiplied alpha fringe:** RGB가 남고 alpha가 0인 source edge pixel을 제거한 버전과 비교한다.
4. **texmap 좌표 경계:** rotation을 0으로 만들었을 때 사라지는지 확인한다.
5. **Lottie layer 구성:** image layer를 한 장씩 꺼서 최초로 artifact를 만드는 layer를 찾는다.

## 수정 방향 계획

1. 원본 `sample.json`과 포함 image를 작업 입력으로 확보하고 frame, canvas size, artifact bounding box를 기록한다.
2. SW/GL/WG를 동일 color space와 viewport로 렌더해 pixel diff heatmap을 만든다.
3. layer → mask/matte → transform → filter 순서로 제거해 최소 asset을 만든다.
4. `SwRenderer::renderImage()`에서 선택된 direct/scaled/texmap/RLE 분기를 로그로 고정한다.
5. 최초 차이를 만드는 sampler 또는 composition 함수에 작은 raw pixel unit test를 추가한 뒤 수정한다.
6. 원본 전체 frame과 synthetic edge fixture를 함께 회귀 테스트한다.

## 실현 가능성 판단

코드 경로를 단계적으로 격리할 수 있어 해결 불가능한 문제는 아니다. 그러나 로컬에 재현 입력이 없는 현재 상태에서는 수정 지점을 고르는 것이 불가능하다. asset과 정확한 frame/좌표가 제공되면 실현 가능성은 중간, 현재 입력만으로는 **중간 이하**다.

## 위험/검증

- sampler의 반 픽셀 offset 변경은 정상 이미지 전체를 한 픽셀 미만으로 흔들 수 있다.
- down-sampling 품질 개선은 CPU 비용을 크게 올릴 수 있다.
- premultiplied/straight alpha를 혼동하면 어두운 halo나 색 번짐이 생긴다.
- 32-bit color와 8-bit mask surface를 모두 검증해야 한다.
- partial rendering과 full rendering이 같은 픽셀을 만드는지 확인한다.
