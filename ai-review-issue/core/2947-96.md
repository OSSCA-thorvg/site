## 이슈 요약

현재 affine으로 정의된 3×3 `Matrix`를 projective 2D(2.5D) 변환에 쓰자는 기능 요청이다. 저장 공간은 이미 9개 원소지만 수학 utility, geometry, bounds, clip과 backend가 affine을 전제로 하므로 전체 파이프라인 기능이다.

## main 코드 조사

**확인된 증거**

- `inc/thorvg.h`는 `e31=e32=0`, `e33=1`인 affine Matrix라고 명시한다.
- Matrix 곱셈/inverse는 3×3 전체를 계산하지만 `Point * Matrix`는 `w`를 계산하거나 나누지 않는다.
- SW outline/bounds가 이 Point 연산을 사용하므로 마지막 행을 무시한다.
- GL은 3×3 마지막 행까지 upload하지만 vertex shader가 `glPos.xy`만 쓰고 `gl_Position.w=1.0`으로 설정해 projective 성분을 버린다.
- WG의 4×4 변환에는 `e31/e32/e33`가 들어가 clip position의 w에 도달할 여지가 있으나 geometry/bounds 공통 전제가 affine이라 부분 지원일 뿐이다.

```cpp
// src/common/tvgMath.cpp: 현재 affine point transform
auto tx = pt.x * m.e11 + pt.y * m.e12 + m.e13;
auto ty = pt.x * m.e21 + pt.y * m.e22 + m.e23;
return {tx, ty};  // w와 perspective divide가 없다.
```

```text
projective: x'=(e11*x+e12*y+e13)/w
            y'=(e21*x+e22*y+e23)/w
            w = e31*x+e32*y+e33
```

| 영역 | 현재 projective 상태 |
|---|---|
| SW | Point 변환이 마지막 행을 무시함 |
| GL | 행은 upload하지만 shader가 w=1로 고정함 |
| WG | clip w에 전달 가능하나 CPU geometry/bounds와 불일치 |
| bounds/intersects | affine corner/path 전제 |

## 원인 가설과 확인 방법

- **확정:** public contract와 공통 Point 연산이 affine이며 GL도 projective divide를 사용하지 않는다.
- **가설:** Matrix 저장 공간만 활용하면 기능을 켤 수 있다는 전제는 틀리고, 곡선/clip/bounds 정책 부재가 실제 차단점이다.
- **확인 방법:** 사각형 하나의 projective matrix를 공통 fixture로 만들고 transformed geometry, backend pixel, bounds/intersects가 각각 어디서 달라지는지 단계별 비교한다.

## 수정 방향 계획

1. matrix convention, API opt-in, `w≈0`, `w<0`, viewport clipping을 먼저 명세한다.
2. Point API에 무조건 divide를 넣어 기존 내부 호출을 바꾸기보다 affine/projective 경로를 명시적으로 분리한다.
3. line/curve를 projective 공간에서 어느 tolerance로 tessellate할지 공통 geometry 층에서 설계한다.
4. Shape, stroke, image, gradient, mask/clip, bounds/intersects를 SW부터 vertical slice로 만들고 GL/WG를 맞춘다.
5. C++/CAPI 문서와 기존 affine bit-for-bit regression을 추가한다.

## 실현 가능성 판단

연구 prototype은 가능하지만 전체 이슈는 **낮음**이다. API 의미와 tessellation 정책이 먼저 결정되어야 하며, 단일 backend 구현은 완료로 볼 수 없다. 초심자는 수학 fixture나 affine regression test만 하위 작업으로 맡는 편이 좋다.

## 위험/검증

- `w≈0`, behind-camera geometry, 큰 coordinate와 무한/NaN bounds를 검사한다.
- SW/GL/WG의 shape/image/gradient/clip 결과와 hit-test를 같은 matrix로 비교한다.
- affine matrix에서는 기존 pixel/performance가 바뀌지 않아야 한다.
