# #3895 Any plan for implementing RGB888-only colour space?

- Link: https://github.com/thorvg/thorvg/issues/3895
- 난이도: 92/100
- 실현 가능성: 낮음
- 초심자 추천: 비추천
- 관련 영역: ColorSpace API, SW raster/blend, SIMD, embedded framebuffer
- 배울 수 있는 것: packed pixel, alpha composition, stride/alignment, conversion strategy

## 이슈 요약

e-paper/embedded framebuffer를 위해 RGB888/RGB565 같은 alpha 없는 packed output을 지원하자는 요청이다. 최종 output alpha 제거와 내부 composition alpha 제거는 다른 문제이며 native 3-byte raster는 엔진 전반의 32-bit 가정을 바꾼다.

## main 코드 조사

**확인된 증거**

- `ColorSpace`는 네 32-bit format과 `Grayscale8`만 제공한다.
- `CHANNEL_SIZE()`는 4 또는 1만 반환한다.
- SW raster/AVX/NEON은 대체로 `uint32_t`와 `uint8_t` 두 경로이며 composition은 alpha를 사용한다.
- GL/WG Canvas target은 더 제한된 32-bit format만 받으므로 이 요청의 실질 MVP는 SW output이다.

```cpp
switch (cs) {
    case ColorSpace::ABGR8888: /* ... */ return sizeof(uint32_t);
    case ColorSpace::Grayscale8: return sizeof(uint8_t);
    default: return 0;
}
```

| 전략 | RAM output | 내부 변경 | 예상 특성 |
|---|---:|---|---|
| 32-bit render 후 RGB convert | 3/2 Bpp final | 작음 | extra pass 필요 |
| native RGB888 raster | 3 Bpp | 매우 큼 | unaligned/SIMD 불리 가능 |
| native RGB565 raster | 2 Bpp | 큼 | 색 정밀도 손실 |

## 원인 가설과 확인 방법

- **확정:** 3/2-byte surface 경로가 없다.
- **가설:** native RGB888이 RAM을 줄여도 unaligned access 때문에 rendering이 빨라지지 않을 수 있다.
- **확인 방법:** 동일 embedded-sized frame에서 32-bit+convert와 native prototype의 peak memory, bandwidth, frame time을 측정한다.

## 수정 방향 계획

1. RGB888 또는 RGB565 하나와 opaque background/alpha discard 규칙을 결정한다.
2. 먼저 32-bit internal compositor + final conversion MVP로 API/use case를 검증한다.
3. memory가 여전히 문제일 때만 clear/solid/image부터 native packed raster를 단계적으로 추가한다.
4. ColorSpace/CAPI enum과 Canvas target validation을 ABI review한다.

## 실현 가능성 판단

최종 conversion MVP는 가능하지만 원 이슈 전체 native 지원은 **낮음**이다. 초심자용이 아니다.

## 위험/검증

- endian, odd stride/width, unaligned access, transparent input의 background와 RGB565 rounding을 검사한다.
- scalar/AVX/NEON 결과 일치와 embedded benchmark를 확인한다.
- mask/effect intermediate는 alpha format을 유지하는지 명시한다.
