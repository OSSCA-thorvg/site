# #3230 — 폐곡선 stroke join의 작은 seam

- Link: https://github.com/thorvg/thorvg/issues/3230
- 난이도: 70/100
- 실현 가능성: 중간
- 초심자 추천: 조건부 (순수 Shape 재현까지)
- 분석 기준: `main` working tree `f989b27892ba`
- 관련 영역: common RenderPath, CPU stroke, GL/WG stroker
- 배울 수 있는 것: tangent/orientation, closed contour join, tolerance와 scale

## 이슈 요약

dash/trim 없는 Lottie loop의 왼쪽 closure에서 작은 seam이 보이며 모든 engine에 존재하지만 GPU에서 더 뚜렷하다는 이슈다. GPU 두 stroker는 마지막 점과 첫 점이 1/64 이내면 closing line을 생략하고 첫 방향으로 join한다. CPU는 별도 outline stroker를 쓴다. 모든 backend에서 보인다는 점 때문에 Lottie가 생성한 nearly-closed path/tangent 또는 공통 입력의 수치 문제가 우선 후보지만 attachment가 local에 없어 확정하지 않는다.

## main 코드 조사

### 확인된 사실

- GL과 WG stroker의 `close()`는 동일하게 첫/마지막 점 거리 `> 0.015625f`일 때만 `lineTo(firstPt)`를 추가하고, 이후 `join(firstPtDir)`를 호출한다.
- `0.015625f`는 `1/64`이며 object 좌표에 고정된 tolerance다. transform scale에 따른 device pixel 크기와 직접 연동되지 않는다.
- GPU `lineTo()`는 zero direction을 바로 반환하고 첫 유효 segment에서 `firstPtDir`을 정한다.
- CPU [`_endSubPath()`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwStroke.cpp)은 closed path면 필요한 closing line 후 incoming/outgoing angle로 inside/outside join을 만든다.
- issue가 모든 engine에서 보인다는 관찰은 GPU-only MSAA보다 input endpoint/tangent 불일치 가능성을 높이지만 확정 증거는 아니다.

```cpp
// GL/WG 공통 closure 기준
if (length(mState.prevPt - mState.firstPt) > 0.015625f) {
    lineTo(mState.firstPt);
}
join(mState.firstPtDir);
```

```text
last ●──── distance <= 1/64 ────● first
     closing segment 생략
     그러나 prevPtDir과 firstPtDir로 join 생성
     → endpoint/tangent가 정확히 호환되지 않으면 seam 후보
```

### 아직 가설인 부분

- **가설 A:** 끝점이 nearly equal이라 closing line은 생략되지만 tangent가 불연속이라 join geometry에 작은 gap/돌기가 생긴다.
- **가설 B:** Lottie path가 첫/끝 좌표를 정확히 normalize하지 않아 세 backend 모두 영향을 받는다.
- **가설 C:** CPU의 미세한 seam과 GPU의 큰 seam이 같은 input 원인에 서로 다른 AA/tessellation이 더해진 결과일 수 있다.

## 수정 방향과 실현 가능성

1. attachment의 최종 `RenderPath` command/point를 dump해 순수 `Shape` fixture로 고정한다.
2. first/last distance, incoming/outgoing tangent와 generated join vertices를 세 backend에서 비교한다.
3. exact-close, `0.5/1/2 × 1/64` offset과 여러 scale을 sweep한다.
4. loader/common 단계에서 endpoint normalize할지 backend tolerance를 device scale에 맞출지 결정한다.
5. truly open path의 cap을 없애지 않는지 round/bevel/miter와 CPU/GL/WG pixel diff를 검사한다.

**판정:** 최소 Shape 재현은 초심자에게 적합하다. tolerance를 바로 바꾸는 patch는 open path 회귀 위험 때문에 충분한 matrix test가 필요하다.
