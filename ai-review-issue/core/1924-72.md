# #1924 — A8 렌더 타깃 지원

- **Link:** https://github.com/thorvg/thorvg/issues/1924
- **난이도:** 72/100
- **초심자 추천:** 조건부(8비트 내부 경로 테스트부터 권장)
- **관련 영역:** `ColorSpace`, SW target API/C ABI, 8비트 래스터·합성
- **배울 수 있는 것:** 픽셀 포맷, stride, alpha coverage, 내부 포맷과 공개 API의 경계
- **조사 기준:** `main@f989b27892bab31f224f810a54782055eba1e3bc`

## 이슈 요약

임베디드 환경에서 32비트 RGBA 대신 픽셀당 1바이트 A8 출력 타깃을 제공해 메모리와 clear/fill 비용을 줄이자는 요청이다. 현재 main에는 과거 문서가 가정한 것보다 많은 8비트 기반 코드가 이미 들어와 있으므로, 남은 문제는 “새 래스터 전체”보다 “공개 target으로 안전하게 노출하고 완성도를 검증”하는 쪽에 가깝다.

## main 코드 조사

### 확인된 증거

- C++ `ColorSpace::Grayscale8`와 C `TVG_COLORSPACE_GRAYSCALE8`가 이미 존재한다. C enum 주석은 since 1.1이라고 적혀 있다.
- `RenderSurface`는 `buf32`/`buf8` union과 `channelSize`를 가지며 `CHANNEL_SIZE(Grayscale8)`은 1을 반환한다.
- `tvgSwRasterC.h`, AVX, NEON 및 `tvgSwRaster.cpp`에는 `channelSize == sizeof(uint8_t)`인 fill/blend/clear 경로가 다수 있다.
- 그러나 `rasterCompositor()`는 ABGR/ARGB 네 형식만 초기화하고 그 밖의 값, 즉 `Grayscale8`도 `NonSupport`로 반환한다.
- `SwCanvas::target()`과 `tvg_swcanvas_set_target()`은 여전히 `uint32_t*`를 받고 문서도 네 32비트 포맷만 지원한다고 명시한다.

```cpp
// tvgRender.h
union {
    pixel_t* data;
    uint32_t* buf32;
    uint8_t* buf8;
};

// tvgSwRaster.cpp: 현재 Grayscale8 target이 여기서 거부된다.
} else {
    return Result::NonSupport;
}
```

### 아직 확인되지 않은 부분

- 모든 image, gradient, mask, blend, post-effect 조합이 외부 `Grayscale8` surface에서도 정확한지는 실행 검증하지 않았다.
- 이슈의 A8이 “source color의 alpha”, “shape coverage”, “흰색을 그린 grayscale” 중 어느 계약인지는 본문만으로 확정되지 않는다.

## 원인 가설

- **확인됨:** 내부 mask용 8비트 surface 지원이 public output보다 먼저 도입됐고 최종 compositor 초기화가 외부 target을 막는다.
- **가설:** 단순히 `rasterCompositor()`에 한 case를 추가하면 solid shape는 동작할 가능성이 높지만, `join`/luma 함수가 필요한 일부 경로에서 null 함수 포인터 또는 잘못된 의미가 드러날 수 있다.
- **설계 쟁점:** 기존 `uint32_t*` signature로 1바이트 버퍼를 넘기게 하는 것은 타입·정렬 계약이 나쁘다. C++ overload나 별도 target API가 더 안전하지만 API/ABI 검토가 필요하다.

```text
32-bit target: uint32_t* -> channelSize=4 -> ABGR/ARGB join -> RGBA output
A8 target:      uint8_t* -> channelSize=1 -> coverage/alpha only -> A8 output
                         ^ 현재 public signature와 compositor 연결이 끊김
```

## 수정 방향과 실현 가능성

1. A8의 값 의미와 stride 단위를 API 문서로 먼저 결정한다.
2. 내부 테스트에서 `RenderSurface{Grayscale8}`로 solid/translucent shape, image, gradient, clip/mask를 실행해 이미 되는 경로와 실패 경로를 목록화한다.
3. `rasterCompositor()`의 8비트 초기화에 필요한 함수 집합을 분리하고, 색상 채널을 읽는 기능은 명시적으로 금지하거나 alpha 의미로 구현한다.
4. C++에는 `uint8_t*` overload/새 함수, C에는 호환 가능한 별도 함수 또는 `void*` API를 비교한다. 기존 symbol/layout은 유지한다.
5. 32비트 ABGR 출력의 alpha 채널과 A8 결과를 bit-exact 비교하고, 실제 LVGL 크기로 RAM·clear·draw 시간을 측정한다.

## 위험과 검증

- enum이 이미 공개돼 있어 이름을 A8로 바꾸는 것은 ABI/source compatibility 문제가 된다. 별칭 또는 의미 문서화가 현실적이다.
- `stride`가 pixel 단위인지 byte 단위인지 반드시 고정해야 한다.
- GL/WG는 이 이슈 범위에서 `NonSupport`를 명확히 유지하는 CPU-only 정책이 가장 작은 구현이다.
