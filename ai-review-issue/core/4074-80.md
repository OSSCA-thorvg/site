## 이슈 요약

같은 fill/opacity/blend context의 여러 shape polygon을 모아 한 draw call로 그려 GPU state change와 호출 수를 줄이자는 최적화 umbrella다.

## main 코드 조사

- 현재 GL에는 `src/renderer/gpu_engine/gl/tvgGlSolidBatch.h`와 `src/renderer/gpu_engine/gl/tvgGlSolidBatch.cpp`가 있고 조건이 맞는 solid shape를 같은 pass/task에 append한다.
- `src/renderer/gpu_engine/gl/tvgGlStencilCoverBatch.h`와 `src/renderer/gpu_engine/gl/tvgGlStencilCoverBatch.cpp`도 stencil/cover geometry batching을 담당한다.
- clip chain, blend, pass boundary 등은 hard batch boundary로 처리된다.
- WG compositor는 shape마다 pipeline/bind group과 `wgpuRenderPassEncoderDrawIndexed()`를 호출하는 구조가 남아 있다.
- 이슈의 두 sub-issue는 체크되어 있어 원래 범위 중 무엇이 남았는지 acceptance criterion이 갱신되어야 한다.

```text
안전한 후보:  A(red, normal, same pass) -> B(red, normal, same pass)
               [인접하고 state/order가 같을 때만 vertex/index append]

hard boundary: clip/mask | blend 변경 | pass 변경 | 교차 bounds | buffer cap
```

| 항목 | GL current main | WG current main |
|---|---|---|
| solid batching | `GlSolidBatch` 있음 | shape별 render 경로 |
| stencil/cover merge | `GlStencilCoverBatch` 있음 | compositor가 shape별 stencil/cover draw |
| shared staging | `GlStageBuffer` | `WgStageBufferGeometry`가 존재하지만 draw command는 개별 mesh 중심 |

## 원인 가설

**추론:** GL 제안은 상당 부분 구현됐고 남은 후보는 WG batching, gradient/image, 더 넓은 stencil case다. 무작정 context 정렬하면 painter's order와 blending을 깨뜨리므로 인접하고 교환 안전한 draw만 합쳐야 한다.

## 수정 방향과 실현 가능성

1. GL/WG 장면별 draw-call baseline과 CPU submission time을 측정한다.
2. 현재 GL batch 조건을 문서화하고 WG에 적용 가능한 최소 solid 인접 batch를 설계한다.
3. 실제 수치 이득이 있는 경우에만 범위를 확장한다.
4. umbrella가 이미 충족됐다면 benchmark와 구현 mapping을 남기고 닫고, 남은 WG 범위는 acceptance가 좁은 새 이슈로 분리한다.

## 위험/검증

depth, clip/mask, opacity, blend, partial redraw, buffer cap과 WASM memory가 핵심이다. 성능 주장은 GPU별 측정이 필요하며 초심자 범위를 넘는다.
