## 이슈 요약

Windows 11의 MSYS2 CLANG64 환경에서 WG backend로 창의 왼쪽 절반만 그리면, 그리지 않은 오른쪽 절반의 alpha `0`이 native compositor까지 전달되어 바탕화면이 비친다. SW와 GL에서는 같은 영역이 opaque black으로 보인다.

보고자가 [`34704ae`](https://github.com/thorvg/thorvg/commit/34704aed8425a3071b951588ee203be010dcaed5)를 전후로 비교한 결과, Windows에서는 `WGPUCompositeAlphaMode_Premultiplied`가 선택되고 WSL에서는 요청 모드가 없어 `Auto`에 머물렀다. maintainer도 동작을 재현했으며, 현재 투명 surface 기능 자체는 유지하려는 입장을 밝혔다.

## main 코드 조사

현재 [`WgRenderer::surfaceConfigure()`](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/src/renderer/gpu_engine/wg/tvgWgRenderer.cpp#L99-L139)은 target color space가 `ABGR8888`인지로 premultiplied 여부를 정한 뒤, surface capability에 같은 composite mode가 있으면 그대로 설정한다.

```cpp
auto premultiplied = (cs == ColorSpace::ABGR8888);
auto alphaMode = premultiplied
               ? WGPUCompositeAlphaMode_Premultiplied
               : WGPUCompositeAlphaMode_Unpremultiplied;
```

그러나 color space는 texture pixel의 RGB/alpha 표현을 설명하고, composite alpha mode는 frame을 OS의 다른 내용과 어떻게 합성할지를 설명한다. WebGPU native header도 `Opaque`에서는 alpha를 1로 취급하고, `Auto`는 구현이 `Opaque` 또는 `Inherit` 중 선택한다고 정의한다.

public [`WgCanvas::Context`](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/inc/thorvg.h#L2448-L2504)에는 instance, adapter, device만 있고 composition 의도를 전달할 필드가 없다. adapter를 생략하는 구형 target 경로는 [`Context.adapter = nullptr`](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/src/renderer/tvgCanvas.cpp#L226-L236)로 전달돼 capability 기반 alpha mode 선택을 건너뛴다.

## 원인 가설

- **확인:** adapter가 있고 요청 alpha mode가 지원되면 미도색 pixel의 alpha `0`이 native composition에 반영된다.
- **확인:** adapter가 없으면 current code는 명시적 선택을 건너뛰고 기본 `Auto`를 사용한다.
- **추론:** 결함의 핵심은 premultiplication 자체가 아니라, pixel representation을 surface 투명성 의도의 대리 변수로 사용한 API 계약이다.
- **정책 미정:** maintainer 답변상 투명 창은 의도된 기능이므로 모든 surface를 `Opaque`로 바꾸는 수정은 다른 사용 사례를 회귀시킨다.

## 수정 방향 계획

1. `WgCanvas::Context` 또는 `target()`에 `Auto`/`Opaque`/`Premultiplied`/`Unpremultiplied`를 표현하는 composition option을 추가한다.
2. 기존 overload의 기본값은 호환 정책에 따라 `Auto`로 두고, 새 option을 준 경우에만 capability를 검증해 명시 모드를 선택한다.
3. color space는 blit shader의 straight/premultiplied 변환에만 사용하고 native window composition 선택과 분리한다.
4. 요청 모드가 capability에 없을 때의 반환값 또는 fallback을 문서화한다. 조용히 다른 투명성 정책으로 바뀌지 않게 한다.
5. API 확정 전에는 이슈 댓글의 두 우회법 — adapter 생략으로 `Auto` 사용, 또는 full-window opaque background 추가 — 를 문서에 명시한다.

## 초심자 시작 가이드

코드를 바꾸기 전에 다음 네 case의 기대 결과를 표로 고정하는 것이 좋다.

| pixel color space | composition 요청 | 미도색 alpha | 기대 창 결과 |
|---|---|---:|---|
| ABGR8888 | Opaque | 0 | 불투명 |
| ABGR8888 | Premultiplied | 0 | 투명 |
| ABGR8888S | Opaque | 0 | 불투명 |
| ABGR8888S | Unpremultiplied | 0 | 투명 |

이후 `surfaceConfigure()`의 mode 선택을 작은 helper로 분리하면 capability 배열을 넣는 단위 테스트를 작성하기 쉽다.

## 위험/검증

- Windows MSYS2/Dawn, WSL/Linux, macOS/iOS와 Emscripten에서 surface를 실제 present한다.
- adapter 유무와 `Auto`, `Opaque`, 두 translucent mode를 각각 검사한다.
- `ABGR8888`과 `ABGR8888S`에서 색상과 투명도가 모두 맞는지 확인한다.
- 미도색 영역, 반투명 shape, 완전 불투명 background를 별도 case로 둔다.
- 지원하지 않는 명시 mode의 fallback·오류 동작과 기존 target overload의 source/ABI 호환을 확인한다.
