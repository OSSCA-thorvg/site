# Issue #4580 — 변경되지 않은 Lottie image asset의 반복 texture upload

- 관련 영역: Lottie animation invalidation, `RenderUpdateFlag`, GL/WG texture cache, dynamic bitmap
- 배울 수 있는 것: dirty tracking, resource revision, shared GPU cache, CPU→GPU upload profiling

## 이슈 요약

큰 embedded image를 포함한 Lottie를 여러 개 재생하면 매 frame GL과 WG가 변하지 않은 image asset까지 다시 upload해 FPS가 떨어진다. GL은 bitmap 갱신 누락을 고친 [`58da8f0`](https://github.com/thorvg/thorvg/commit/58da8f0cbc4085ee9b95c1cd83991bcea19de1c0) 이후 `Image` flag마다 `glTexImage2D()`를 호출한다. WG도 같은 flag를 texture refresh로 해석한다.

## main 코드 조사

[`Animation::frame()`](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/src/renderer/tvgAnimation.cpp#L38-L49)은 loader가 frame을 바꾸면 root picture를 `RenderUpdateFlag::All`로 표시한다. [`All`](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/src/renderer/tvgRender.h#L43-L45)에는 `Image`가 포함된다.

Lottie image는 parse 후 [`LottieImage::prepare()`](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/src/loaders/lottie/tvgLottieModel.cpp#L298-L310)에서 한 번 Picture로 load된다. builder는 [`updateImage()`](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/src/loaders/lottie/tvgLottieBuilder.cpp#L964-L985)에서 같은 Picture를 ref 또는 duplicate해 여러 layer가 공유하게 한다.

GL renderer는 surface pointer와 filter가 같아 기존 texture를 재사용하지만, [`Image` flag가 오면 `TextureMgr::upload()`](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/src/renderer/gpu_engine/gl/tvgGlRenderer.cpp#L1281-L1298)를 다시 부른다. 이 함수는 [`glTexImage2D()`로 전체 image를 재정의](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/src/renderer/gpu_engine/gl/tvgGlTextureMgr.cpp#L34-L42)한다.

WG도 [`Image` 또는 `Path` flag를 `refreshTexture`로 변환](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/src/renderer/gpu_engine/wg/tvgWgRenderer.cpp#L195-L215)하고, manager가 [`refreshTexture`이면 `upload()`](https://github.com/thorvg/thorvg/blob/1341c1ebd8c34134f0b4944fe21d192d1c43bb21/src/renderer/gpu_engine/wg/tvgWgTextureMgr.cpp#L82-L104)한다.

## 원인 가설

- **확인:** animation frame 변경이 모든 하위 paint의 image content 변경으로 과대 해석된다.
- **확인:** GL과 WG는 같은 `RenderSurface*`를 cache key로 쓰지만 “마지막으로 upload한 content 세대”는 저장하지 않는다.
- **확인:** embedded Lottie image는 animation frame과 독립적으로 load된 정적 Picture다.
- **추론:** 가장 안전한 수정 지점은 Lottie만 예외 처리하는 renderer 조건문이 아니라 bitmap content revision이다. 그래야 video와 외부 resolver처럼 실제로 pixel이 바뀌는 입력도 보존된다.

## 수정 방향 계획

1. `RenderSurface` 또는 loader-owned bitmap에 단조 증가하는 `contentRevision`을 둔다.
2. bitmap decode, raw data 교체, video frame 갱신과 resolver가 실제 pixel을 바꿀 때만 revision을 올린다.
3. GL/WG texture entry에 마지막 upload revision을 기록하고 `(surface, filter, revision)`이 같으면 upload를 생략한다.
4. 같은 surface를 여러 Picture duplicate가 참조해도 한 revision당 한 번만 upload되도록 manager 수준에서 판정한다.
5. geometry/transform/opacity 변경은 기존 `RenderUpdateFlag`로 처리하되 image content dirty와 분리한다.
6. 장기적으로 `Animation::frame()`이 `All` 대신 loader가 계산한 세부 update mask를 반환할 수 있는지 검토한다.

## 초심자 시작 가이드

먼저 GL의 `TextureMgr::upload()`와 WG의 `WgTextureMgr::upload()` 호출 횟수·byte 수를 frame별로 기록한다. 다음 세 asset을 분리해서 보면 잘못된 최적화를 피할 수 있다.

| asset | frame 변경 시 pixel 변경 | 기대 upload |
|---|---:|---:|
| embedded PNG/JPEG | 아니오 | 최초 1회 |
| 외부 resolver 정적 image | 아니오 | resolve 후 1회 |
| video/animated bitmap | 예 | content revision 변경 때마다 |

`glTexSubImage2D()`로 바꾸는 것만으로는 전송량이 그대로이므로 핵심 해결이 아니다.

## 위험/검증

- 이슈의 2001×2001 image, 50개 animation으로 frame time과 upload byte/call 수를 전후 비교한다.
- 하나의 asset을 여러 layer와 여러 Animation이 공유할 때 중복 upload가 없는지 확인한다.
- video, raw bitmap update, asset resolver가 새 pixel을 제공하면 반드시 화면이 갱신되는지 확인한다.
- texture size·color space·filter 변경과 context/cache reset 뒤에는 upload가 다시 일어나는지 확인한다.
- GL과 WG 결과 image를 CPU backend reference와 비교하고 sanitizer·resource lifetime 검사를 함께 실행한다.
