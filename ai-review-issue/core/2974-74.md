## 이슈 요약

SVG2가 허용하는 바깥쪽 focal circle 등 radial gradient를 GL처럼 SW/WG에서도 렌더링하자는 요청이다. parser는 `fx/fy/fr`를 이미 읽지만 공통 보정 함수가 값을 안쪽으로 clamp하므로, 입력 의미가 rasterizer 전에 바뀐다.

## main 코드 조사

**확인된 증거**

- SVG loader는 `cx/cy/r/fx/fy/fr`를 파싱해 `RadialGradient::radial()`로 전달한다.
- `RadialGradientImpl::correct()`는 focal point를 end circle 안으로 옮기고 `fr`을 `(r-distance)`보다 작게 줄인다.
- 주석은 이 보정을 SW/WG의 SVG2 지원이 되면 제거할 대상으로 명시하며 GL은 이미 지원한다고 기록한다.
- SW `_prepareRadial()`은 `correct()` 결과로 quadratic 계수를 만들고, WG shader는 단순 discriminant/root 공식을 사용한다. GL shader는 radial/strip/focal-on-circle 등을 더 세분한다.

```cpp
// src/renderer/tvgFill.h
if (this->r - dist < PRECISION) { /* focal을 circle 안으로 이동 */ }
auto maxFr = (r - dist) * (1.0f - PRECISION);
fr = (this->fr > maxFr) ? std::max(0.0f, maxFr) : this->fr;
```

| backend | 현재 핵심 동작 |
|---|---|
| SW | `correct()` 후 quadratic, 작은 `a` 보정 |
| WG | `correct()`된 uniform + 한 quadratic root |
| GL | 여러 geometry case를 나눈 shader; 소스 주석상 SVG2 쪽 기준 |

## 원인 가설과 확인 방법

- **확정:** parser 누락이 아니라 공통 `correct()`가 SVG2 입력을 제한한다.
- **가설:** SW/WG가 바깥 focal/`fr>=r`에서 0 division, 잘못된 root 또는 NaN을 만들 수 있어 보정이 도입됐다.
- **확인 방법:** 보정을 끈 진단 build에서 GL과 같은 최소 gradient cases를 SW/WG에 입력하고 coefficient/discriminant/t 결과가 처음 갈리는 case를 찾는다.

## 수정 방향 계획

1. focal inside/outside/on-circle, `fr=0`, `fr=r`, `fr>r`, coincident center를 fixture 표로 만든다.
2. GL shader의 case 분기를 공통 수학 reference로 문서화하고 SW scalar prototype을 먼저 맞춘다.
3. WGSL도 동일 branch/epsilon 의미로 옮기고 NaN/Inf가 uniform/output에 퍼지지 않게 한다.
4. `correct()`를 제거하기 전에 SVG와 Lottie가 다른 compatibility policy를 가져야 하는지 loader 층에서 결정한다.

## 실현 가능성 판단

**중간**이다. 문제 위치와 GL reference가 분명해 단계적으로 접근할 수 있다. 다만 초심자는 fixture와 cross-backend image diff를 맡는 것이 적합하고, 수학/epsilon 수정은 경험자의 검토가 필요하다.

## 위험/검증

- SW/WG/GL과 browser reference를 비교하되 GPU precision tolerance를 둔다.
- 기존 SVG Tiny와 Lottie radial sample이 바뀌지 않는지 별도 regression을 둔다.
- 모든 case에서 NaN/Inf, division by zero, out-of-range texture lookup을 검사한다.
