## 이슈 요약

여러 Lottie 차이를 묶은 umbrella 보고로, 기존 로컬 문서는 대부분 수정됐고 dice의 3D transform 문제가 남았다고 기록한다. 첨부 asset은 현재 workspace에 없으므로 이 판단을 재렌더링으로 독립 검증할 수 없고, current main의 3D lowering 한계만 확인했다.

## main 코드 조사

**현재 소스에서 확인된 증거**

- parser는 3D layer의 `rx/ry/rz/or`를 읽는다.
- `_dimension3d()`는 x/y/z 회전 행렬 중 최종 `e11/e12/e21/e22` 네 affine 성분만 만든다.
- z position, camera, homogeneous w 또는 perspective를 공통 renderer로 전달하는 모델은 없다.
- 따라서 true 3D 장면은 2D affine projection으로 축약된다.

**기존 문서에만 남은 정보**

- 이전 문서는 color/Pucker-Bloat 문제는 수정됐고 dice asset이 residual이라고 기록한다. 첨부 `data.json`/`test.json`은 저장소에 없어 이번 조사에서 다시 확인하지 않았다.

```cpp
// src/loaders/lottie/tvgLottieBuilder.cpp
m.e11 = ri00 * ro00 + ri01 * ro10 + ri02 * ro20;
m.e12 = ri00 * ro01 + ri01 * ro11 + ri02 * ro21;
m.e21 = ri10 * ro00 + ri11 * ro10 + ri12 * ro20;
m.e22 = ri10 * ro01 + ri11 * ro11 + ri12 * ro21;
```

```text
Lottie 3D transform -> 3D rotation 계산 -> 2×2 affine 성분만 보존
camera / z / perspective ----------------X 공통 renderer 전달 없음
```

## 원인 가설과 확인 방법

- **확정:** current main은 Lottie 3D를 완전한 projective scene으로 표현하지 않는다.
- **가설:** 남은 dice 차이는 이 affine 축약 때문에 발생한다. 첨부 없이 확정할 수 없다.
- **확인 방법:** asset을 합법적인 test fixture로 확보하고 한 precomp/한 layer/한 transform으로 축소해 reference matrix와 비교한다.

## 수정 방향 계획

1. 두 asset을 current main과 reference renderer에서 frame별 비교해 이미 해결된 항목을 제거한다.
2. residual을 transform 조합 하나로 축소하고 `_dimension3d()` 출력과 expected screen coordinates를 기록한다.
3. orthographic 수식 보정으로 끝나는지, camera/perspective 지원이 필요한지 design gate를 둔다.
4. perspective가 필요하면 #2947 범위와 합치고 단일 Lottie patch로 우회하지 않는다.

## 실현 가능성 판단

재현 fixture가 없고 residual이 core perspective 기능일 가능성이 있어 **낮음**이다. 초심자는 asset 축소를 맡을 수 있으나 전체 수정은 적합하지 않다.

## 위험/검증

- 기존 2D/3D Lottie transform, precomp parent chain과 orientation을 회귀 검사한다.
- SW/GL/WG screen coordinates와 pixels를 동일 frame에서 비교한다.
- umbrella issue를 서로 다른 원인의 patch 하나로 처리하지 않는다.
