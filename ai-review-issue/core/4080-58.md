## 이슈 요약

path가 바뀌지 않았으면 WebGPU의 기존 fill/stroke mesh를 유지해 tessellation 비용을 줄이자는 최적화다.

## main 코드 조사

- `src/renderer/gpu_engine/wg/tvgWgRenderData.cpp::WgRenderDataShape::updateMeshes()` 첫 줄은 항상 `releaseMeshes()`를 호출하며 “path changes가 없으면 재사용” TODO가 그대로 있다.
- 아래 코드는 `RenderUpdateFlag::Transform | Path`를 `updatePath`로 계산하고 fill/stroke 재생성 조건을 이미 나누지만, 앞에서 모든 mesh를 비워 이 조건의 이점을 잃는다.
- fill은 Color/Gradient flag에도 tessellate되고 stroke는 Stroke/GradientStroke에도 tessellate되어 geometry와 paint state invalidation이 섞여 있다.

```text
현재 updateMeshes()
  releaseMeshes()  ← 모든 flag에서 fill/stroke/bbox/clip 삭제
  ├─ Path|Transform|Color|Gradient       -> fill tessellate
  └─ Path|Transform|Stroke|GradientStroke -> stroke tessellate

목표
  geometry dirty? yes -> 해당 mesh+bbox rebuild
                  no  -> mesh 유지, paint uniform만 update
```

| 변경 | fill mesh | stroke mesh | paint settings |
|---|---|---|---|
| Color/opacity | 재사용 후보 | 재사용 후보 | 갱신 |
| Path | 재생성 | 재생성 | 필요 시 갱신 |
| Stroke width/dash/join | fill 재사용 가능 | 재생성 | 갱신 |
| Transform | 현재 world-space optimize 영향 때문에 먼저 검증 | 동일 | matrix 갱신 |

## 원인 가설

**강한 코드 단서:** 무조건 release가 색/opacity만 바뀐 update에서도 mesh cache를 파괴한다. 다만 `gpuOptimize()`가 matrix를 사용하므로 transform 변화는 단순 bbox update만으로 재사용 가능한지 별도 확인이 필요하다.

## 수정 방향과 실현 가능성

1. 정적 path에서 color/opacity/frame update별 tessellator 호출 수와 시간을 계측한다.
2. Path/Transform/Stroke geometry flag와 Color/Gradient uniform flag를 분리한다.
3. fill/stroke mesh를 각각 필요한 경우에만 clear/rebuild하고 empty→visible 전환을 테스트한다.
4. 첫 PR은 Color/opacity-only update에서 mesh 주소·count와 tessellator 호출 수가 유지되는 범위로 제한한다.
5. Transform reuse는 `gpuOptimize()`/stroker가 어느 좌표계에 geometry를 저장하는지 benchmark와 bbox test 후 별도 확장한다.

## 위험/검증

trim/dash/stroke width/join/transform은 mesh를 무효화한다. stale bbox, stage buffer offset과 pooled render data 재사용을 검사한다. 초심자는 color-only cache부터 제한하면 조건부로 적합하다.
