## 이슈 요약

ARM 장치의 소프트웨어 raster loop를 NEON으로 확장하는 이슈다. current main에는 solid fill과 translucent rect/RLE 구현이 있지만 source pixel 합성과 gradient 등은 scalar다. 하드웨어별 성능뿐 아니라 기존 NEON alpha 수식과 scalar 수식의 정밀도 차이를 먼저 해결해야 하므로 “intrinsic 몇 줄 추가”보다 범위가 크다.

## main 코드 조사

### 확인된 사실

- [`tvgSwRasterNeon.h`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwRasterNeon.h)는 `neonRasterGrayscale8`, `neonRasterPixel32`, translucent rect/RLE 네 경로를 제공한다.
- AArch64 solid fill은 `vst4q_u32`, AArch32는 `vst1q_u32`를 사용하며 서로 다른 loop 단위를 가진다.
- NEON의 vector `ALPHA_BLEND()`는 16-bit 곱의 하위 8bit를 버려 사실상 256으로 나눈다. scalar [`ALPHA_BLEND()`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwCommon.h)는 bit mask와 보정으로 `/255`에 가까운 결과를 만든다.
- Windows ARM 지원 TODO가 header에 남아 있다.
- [`tvgSwRaster.cpp`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwRaster.cpp)의 source-array pixel 합성 두 함수에는 SIMD TODO가 있으며 NEON dispatch가 없다.

정밀도 차이가 드러나는 핵심 코드는 다음과 같다.

```cpp
// NEON: (c * a) >> 8
static inline uint8x8_t ALPHA_BLEND(uint8x8_t c, uint8x8_t a)
{
    uint16x8_t t = vmull_u8(c, a);
    return vshrn_n_u16(t, 8);
}
```

```text
RGBA uint32_t
   └─ reinterpret as 8 x uint8 lanes
        ├─ vmull_u8: 8 x uint16 products
        ├─ shift/narrow
        └─ two pixels written together + scalar leftover
```

### 아직 가설인 부분

- **가설 A:** scalar와 NEON의 1 LSB 차이가 반투명 레이어 반복 합성에서 누적될 수 있다. 무작위·반복 합성 비교 전에는 화면상 회귀 크기를 확정할 수 없다.
- **가설 B:** `rasterTranslucentPixel32()`가 가장 자연스러운 다음 후보지만 실제 ARM profile상 최우선 hot path인지는 확인되지 않았다.
- **가설 C:** Windows ARM TODO는 compile guard뿐 아니라 intrinsic header/compiler 차이가 원인일 수 있다. 로컬 Linux source만으로 지원 범위를 확정할 수 없다.

## 수정 방향과 실현 가능성

1. scalar와 현행 NEON 함수에 동일한 무작위 pixel/coverage를 입력하는 bit-exact 또는 허용 오차 test를 ARM에서 실행한다.
2. `/255` 규약을 먼저 합의한다. 기존 NEON 결과 변경은 별도 correctness patch로 분리하는 편이 안전하다.
3. profile로 고른 함수 하나에만 vector body와 scalar tail을 구현한다.
4. AArch64와 AArch32 compile/run을 각각 확인하고 비정렬 buffer와 짧은 span을 포함한다.
5. Android/Linux ARM에서 결과와 CPU time을 측정하고 Windows ARM은 최소 compile guard를 확인한다.

**판정:** 실제 ARM 장비와 SIMD 경험이 있다면 작은 하위 과제는 가능하다. 장비 없이 cross-compile 성공만으로는 렌더링 정확성과 성능을 입증할 수 없어 전체 이슈의 실현 가능성은 낮다.

## 위험과 검증 체크리스트

- alpha `0/1/127/254/255`와 반복 source-over
- ABGR/ARGB channel order, premultiplied alpha
- len `0/1/2/7/8/15/16/17`과 비정렬 pointer
- AArch32, AArch64, Android NDK, Windows ARM compile guard
- sanitizer 및 scalar fallback build
- 실제 device에서의 CPU time, thermal throttling을 고려한 반복 측정
