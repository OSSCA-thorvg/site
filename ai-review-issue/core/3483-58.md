## 이슈 요약

dasharray `.0357618`과 `.0357619` 같은 극소 차이에서 circle 전체가 갑자기 사라지는 문제다. SVG parser보다 SW의 cubic dash decomposition과 이후 tiny-stroke filtering이 유력하지만 최초 손실 단계는 trace로 확인해야 한다.

## main 코드 조사

### 확인된 증거

- `ShapeImpl::strokeDash()`는 음수를 0으로 만들고 pattern 합을 저장하며 극소 positive 값을 거부하지 않는다.
- `_dashLineTo()`/`_dashCubicTo()`는 `DASH_PATTERN_THRESHOLD=0.001`로 split loop를 종료한다.
- segment 처리 뒤 phase를 넘기는 절대 조건이 line은 `curLen < 1.0f`, cubic은 `< 0.1f`로 서로 다르며 이슈 dash 값보다 크다.
- dashed outline은 `strokeParseOutline()`로 넘어가고 `_tiny()`는 `2/64`보다 작은 delta를 작은 점으로 본다.

```cpp
// 서로 다른 절대 cutoff
if (dash.curLen < 1.0f && !tvg::zero(len)) { /* line */ }
if (dash.curLen < 0.1f && !tvg::zero(len)) { /* cubic */ }

constexpr float EPSILON = 2.0f / 64.0f;  // stroke tiny 판정
```

### 아직 확인되지 않은 부분

- 현재 main에서 두 숫자의 exact pixel 결과와 최초 0 command/border 단계는 실행 확인하지 않았다.
- zoom/transform이 dash pattern 자체에 적용되는 시점과 기대 device-space policy를 fixture로 고정해야 한다.

## 원인 가설

- **강한 가설:** curve split의 phase 누적 오차와 0.1 absolute cutoff가 특정 값에서 on/off 상태를 뒤집는다.
- **후속 가설:** 생성된 dash가 있더라도 `_tiny()`에서 border point가 합쳐져 최종 outline이 비게 될 수 있다.
- **미확인:** 두 단계 중 어느 것이 최초 원인인지 현재 코드 읽기만으로 단정하지 않는다.

```text
SVG dash -> _genDashOutline -> commands/points -> strokeParseOutline -> borders -> RLE
                후보 1                           후보 2
```

## 수정 방향과 실현 가능성

1. `.03570...03582` sweep와 zoom sweep를 자동 regression으로 만들고 visible span count를 기록한다.
2. dash 단계의 command/point count와 stroke 단계 border count를 동시에 trace한다.
3. 최초 0이 되는 cutoff만 pattern/curve length에 상대적인 정책으로 바꾸는 prototype을 비교한다.
4. zero dash를 건너뛰면서 loop가 반드시 진전한다는 invariant를 test한다.
5. line/cubic/close, odd pattern, offset, Butt/Round cap과 SW/GL/WG 결과를 비교한다.

## 위험과 검증

- threshold를 제거하면 극소 pattern에서 iteration 폭증 또는 infinite loop가 생길 수 있다.
- 정확한 pixel 동일보다 “작은 입력 변화에 전체 shape가 사라지지 않음”을 우선 invariant로 둔다.
- 정상 크기 dash 성능과 output도 회귀 검사한다.
