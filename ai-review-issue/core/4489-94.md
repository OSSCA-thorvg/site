## 이슈 요약

Linux media playback에 GStreamer 대신 FFmpeg를 쓰자는 제안이다. 단순 bug fix가 아니라 dependency와 플랫폼 전략을 정해야 하는 설계 논의다.

## main 코드 조사

현재 commit의 `inc/thorvg.h`, `src/loaders/meson.build`, `src/loaders/`에는 공개 media playback API나 `media` loader가 없다. 즉 Issue가 언급한 구현은 현재 main에 아직 없거나 별도 실험 브랜치/후속 계획을 가리키는 것으로 보인다.

```text
현재 main: Picture/Animation -> 정적 image·Lottie loader
제안 범위: Media API -> demux -> video decoder -> timed frames -> Picture/Surface
                         \-> audio/seek/HW decode?  (요구사항 미정)
```

## 원인 가설

확인할 코드가 main에 없으므로 기술적 root cause가 있는 Issue가 아니다. GStreamer의 크기와 FFmpeg의 video-only 구현 범위를 비교하려는 architecture proposal이다.

## 수정 방향 계획

1. maintainers에게 목표 범위(영상만/음성 포함, seek, HW decode, Windows/macOS)를 먼저 확인한다.
2. plugin interface와 dependency를 core binary에서 분리할 수 있는지 설계한다.
3. GStreamer/FFmpeg 각각의 binary size, startup, license, cross-build 결과를 측정한다.

## 위험/검증

플랫폼별 codec 지원과 license, threading, frame ownership 영향이 크다. 구체 요구사항과 main 구현이 생기기 전에는 초심자 단독 작업으로 권하지 않는다.
