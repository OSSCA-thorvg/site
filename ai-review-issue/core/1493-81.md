# #1493 — renderer: optimize paint task scheduling

- Link: https://github.com/thorvg/thorvg/issues/1493
- 난이도: 81/100
- 실현 가능성: 중간 (현행 재현·이슈 재정의), 낮음 (scheduler redesign)
- 초심자 추천: 비추천
- 분석 기준: `main` working tree `f989b27892ba`
- 관련 영역: Canvas state machine, loader/render task, threading
- 배울 수 있는 것: deferred update, work queue, dependency wait, paint ownership과 cancellation

## 이슈 요약

옛 `canvas->push()`가 같은 thread에서 즉시 paint update를 수행하고 Picture loading을 기다리는 병목을 scheduler로 넘기자는 성능 제안이다. current main의 API는 `Canvas::add()`이며 이 함수는 scene에 paint를 넣고 상태만 변경한다. 실제 update traversal은 `update()` 또는 `draw()`에서 일어난다. 따라서 이슈 본문의 전제는 현재 코드와 그대로 일치하지 않으며, scheduler를 바꾸기 전에 현행 병목을 profile로 다시 정의해야 한다.

## main 코드 조사

### 확인된 사실

- [`Canvas::Impl::add()`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/tvgCanvas.h)은 renderer ownership과 Drawing 상태를 검사하고 `status=Painting`, `scene->add()`만 수행한다. paint tree `update()`를 직접 호출하지 않는다.
- `draw()`는 status가 Painting/Damaged면 `update()`를 부르고, `update()`는 dominant thread에서 `PAINT(scene)->update(...)`를 순회한다.
- [`PictureImpl::update()`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/tvgPicture.h)은 시작에서 `load()`를 호출한다. vector loader가 이미 존재하면 `loader->sync()`를 호출할 수 있어 update 시점 대기 후보는 남아 있다.
- [`TaskScheduler`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/tvgTaskScheduler.h)는 worker queue와 `Task::done()` 대기를 제공한다. threads가 0이면 `run(0)`을 동기 호출한다.
- CPU renderer도 별도 render task를 queue하므로 loader task와 renderer task의 dependency/lifetime을 같이 봐야 한다.

현행 상태 흐름은 이슈 설명과 다르다.

```mermaid
stateDiagram-v2
    [*] --> Synced
    Synced --> Painting: Canvas::add()
    Painting --> Updating: Canvas::update() 또는 draw() 내부 update
    Updating --> Drawing: draw()
    Drawing --> Synced: sync()

    note right of Painting
      add()는 scene 삽입과 상태 변경만 수행
    end note
    note right of Updating
      Picture::load()에서 loader sync 가능
    end note
```

### 아직 가설인 부분

- **확인된 불일치:** “push가 즉시 update한다”는 원래 전제는 current `add()` 구현에 적용되지 않는다.
- **가설 A:** 남은 병목은 `draw/update` 시점의 loader `sync()` 대기일 수 있다.
- **가설 B:** 여러 Picture를 add한 경우 loader들은 이미 background task로 진행되고 있어 현행 구조가 충분히 overlap할 수도 있다. benchmark 없이는 scheduler 변경 필요성을 확정할 수 없다.
- **가설 C:** paint update의 일부를 worker로 옮길 수 있지만 GL/WG context 접근과 Canvas state는 dominant thread에 남겨야 할 수 있다.

## 수정 방향과 실현 가능성

1. 큰 SVG/Lottie Picture 여러 개를 대상으로 `load`, `add`, `update`, `draw`, `sync`와 `Task::done()` 대기 시간을 분리 측정한다.
2. threads `0/1/N` 비교로 loader overlap이 이미 일어나는지 확인한다.
3. 원래 병목이 재현되지 않으면 current code evidence와 수치로 이슈 close/재정의를 제안한다.
4. 재현된다면 paint별 loader dependency와 renderer prepare의 thread-safe/affine 단계를 DAG로 작성한다.
5. 독립 prepare만 queue하고 remove/destruction cancellation, 중복 예약과 sync barrier test를 먼저 만든다.

**판정:** “변경 불필요”를 입증하는 benchmark는 실현 가능하다. 실제 scheduler redesign은 성능 수치와 동시성 검증이 필요한 고난도 작업이다.
