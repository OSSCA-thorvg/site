# #1341 — sw_engine: thick strokes issue

- Link: https://github.com/thorvg/thorvg/issues/1341
- 난이도: 66/100
- 실현 가능성: 중간
- 초심자 추천: 조건부 (pixel probe와 정상 합성 여부 판정부터)

## 이슈 요약

12×12 크기의 닫힌 사각형에 폭 64, opacity 0.5인 흰 stroke와 불투명 blue fill을 적용했을 때 경계가 기대와 다르다는 이슈다. 이 입력에서는 stroke가 path 양쪽으로 32씩 뻗어 shape보다 훨씬 크다. 또한 기본 SVG painting order는 fill 후 stroke이므로 안쪽 반투명 stroke를 통해 blue fill이 보이는 현상 자체는 정상일 수 있다. 실제 버그인지 판단하려면 기대 이미지를 source-over 수학과 browser reference로 먼저 명확히 해야 한다.

## main 코드 조사

### 확인된 사실

- [`tvgSvgBuilder.cpp`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/loaders/svg/tvgSvgBuilder.cpp)의 `_applyStroke()`는 width와 opacity를 각각 `strokeWidth()`와 `strokeFill(..., opacity)`로 전달한다.
- [`tvgSwStroke.cpp`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwStroke.cpp)의 `strokeReset()`은 width의 절반을 border 확장량으로 사용한다. width 64이면 양쪽 32다.
- [`SwRenderer::renderShape()`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwRenderer.cpp)은 `strokeFirst()==false`인 기본값에서 fill을 먼저 raster한 뒤 stroke를 raster한다.
- 같은 파일의 `antialiasing()`은 두꺼운 stroke 아래 fill AA를 끌 수 있지만 `stroke color.a < 255`이면 AA를 유지한다. 이 sample의 0.5 alpha는 그 조건에 해당한다.
- fill RLE와 stroke RLE는 [`tvgSwShape.cpp`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/cpu_engine/tvgSwShape.cpp)에서 별도로 생성된다.

불투명 blue 위에 50% white를 source-over하면 안쪽은 순백이 아니다.

```text
inside stroke:  white(α=.5) over blue(α=1)
RGB ≈ (128, 128, 255), alpha = 1

outside stroke: white(α=.5) over transparent
premultiplied RGB ≈ (128, 128, 128), alpha ≈ .5
```

### backend별 확인 포인트

| backend | geometry/coverage 경로 | 이 sample에서 볼 것 |
|---|---|---|
| SW | stroke outline → 26.6 RLE → 별도 fill/stroke raster | 경계 scanline의 두 RLE coverage와 합성값 |
| GL | GPU stroke geometry + stencil/cover | 같은 좌표의 alpha/RGB가 SW와 같은지 |
| WG | `WgRenderData` stroke + compositor | geometry 폭과 painting order가 같은지 |

### 아직 가설인 부분

- **가설 A:** 결과가 source-over 예상값과 일치한다면 이 이슈는 renderer 버그가 아니라 기대/설명 문제일 수 있다.
- **가설 B:** 안쪽 경계에서 fill AA와 stroke AA가 모두 부분 coverage라면 browser와 다른 fringe가 생길 수 있다.
- **가설 C:** width가 path 크기보다 훨씬 커 join/border가 퇴화할 수 있다. opacity 1.0에서도 geometry가 어긋나는지로 AA/composition 가설과 분리할 수 있다.

## 수정 방향과 실현 가능성

1. current main CPU와 browser reference를 같은 배경색/크기로 렌더하고 중앙·경계·외부 pixel RGBA를 기록한다.
2. stroke opacity `1.0/0.5`, width `1/12/64`, `Shape::order(true/false)`를 비교한다.
3. opacity 1에서도 outline 위치가 틀리면 stroke geometry를, 0.5에서만 다르면 coverage/composition을 추적한다.
4. SW/GL/WG pixel diff로 CPU 전용인지 공통 semantics인지 구분한다.
5. source-over 기대와 불일치하는 최초 단계만 수정하고 golden test를 추가한다.

**판정:** 입력이 작아 진단은 실현 가능하다. 다만 “흰 stroke여야 한다” 같은 시각적 기대만으로 alpha blending 코드를 바꾸면 안 된다.
