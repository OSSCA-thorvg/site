# #3171 — RGB565 16-bit software output

- Link: https://github.com/thorvg/thorvg/issues/3171
- 난이도: 90/100
- 실현 가능성: 낮음
- 초심자 추천: 비추천
- 분석 기준: `main` working tree `f989b27892ba`
- 관련 영역: ColorSpace C++/C ABI, CPU raster hot path, RenderSurface
- 배울 수 있는 것: RGB565 quantization, stride/endian, alpha composition, code-size trade-off

## 이슈 요약

LVGL용으로 만든 16-bit RGB565 software renderer를 ThorVG 구조에 통합할 방법과 수요를 묻는 제안이다. current main의 public target pointer와 CPU raster는 32-bit color 또는 내부 8-bit grayscale을 전제로 한다. RGB565는 alpha가 없으므로 type 하나를 추가하는 것만으로 mask/blend/offscreen composition이 해결되지 않으며, native 16-bit pipeline과 32-bit 내부+최종 변환 중 어떤 목표인지 먼저 정해야 한다.

## main 코드 조사

### 확인된 사실

- public [`ColorSpace`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/inc/thorvg.h)는 네 32-bit ABGR/ARGB variant와 `Grayscale8`만 가진다. C API enum도 동일하다.
- `SwCanvas::target()`과 `tvg_swcanvas_set_target()`은 모두 `uint32_t* buffer`를 받는다.
- [`RenderSurface`](https://github.com/thorvg/thorvg/blob/f989b27892bab31f224f810a54782055eba1e3bc/src/renderer/tvgRender.h)의 union에는 `uint32_t* buf32`, `uint8_t* buf8`만 있고 `CHANNEL_SIZE()`도 4 또는 1만 반환한다.
- CPU engine에서 `channelSize == sizeof(uint32_t/uint8_t)` 분기는 50곳 검색되며 raster 본체 외 AVX/NEON/texmap까지 퍼져 있다.
- intermediate compositor와 mask/blend 수식은 alpha를 사용한다. RGB565 target에 alpha byte가 없다고 내부 alpha 필요까지 사라지는 것은 아니다.

```text
선택 A: native RGB565 everywhere
  API uint16_t* → 모든 raster/blend/mask 재구현 → 낮은 buffer 메모리
                  └─ quantization이 매 합성마다 누적될 위험

선택 B: internal RGBA8888 + final RGB565 conversion
  기존 pipeline → 32-bit offscreen → RGB565 target 변환
                  └─ 변경 작음, peak memory 절감은 제한적
```

RGB565 pack의 개념은 단순하지만 pipeline은 그렇지 않다.

```cpp
uint16_t rgb565 = ((r >> 3) << 11) |
                  ((g >> 2) << 5)  |
                   (b >> 3);
```

### 아직 가설인 부분

- **가설 A:** LVGL의 주 목표가 최종 framebuffer 메모리라면 option B만으로도 충분할 수 있다. 실제 RSS/FPS 요구 수치가 필요하다.
- **가설 B:** native 16-bit raster는 write bandwidth를 줄이지만 반복 unpack/blend/pack과 색 손실로 오히려 느리거나 품질이 낮을 수 있다.
- **가설 C:** RenderSurface 전체 template화는 source duplication과 binary size를 키울 수 있다. ThorVG의 작은 binary 목표상 측정 없는 선택은 부적절하다.

## 수정 방향과 실현 가능성

1. RGB565 byte order, stride 단위, alpha 배경색, premultiplied 의미와 C++/C signature를 명세한다.
2. 기존 32-bit pipeline 끝에서 RGB565로 변환하는 baseline을 구현·측정한다.
3. LVGL scene으로 output buffer, compositor peak RSS, CPU time과 banding을 기록한다.
4. native 16-bit가 꼭 필요할 때 solid rect/RLE 한 vertical slice로 function-table/template/code duplication을 비교한다.
5. mask/blend/image/gradient/SIMD와 malformed stride/endian tests를 단계적으로 확장한다.

**판정:** 작은 prototype은 가능하지만 upstream-ready RGB565 backend는 대형 작업이다. 먼저 baseline 측정으로 native pipeline 필요성을 증명해야 한다.
