# #4027 — static JPEG의 비표준 sampling factor 지원

- **Link:** https://github.com/thorvg/thorvg/issues/4027
- **난이도:** 76/100
- **초심자 추천:** 비추천(fixture 분류·비교 측정은 조건부 추천)
- **관련 영역:** bundled JPEG decoder, MCU/component sampling, static build binary size
- **배울 수 있는 것:** YCbCr subsampling, MCU layout, decoder 호환성·fuzzing
- **조사 기준:** `main@f989b27892bab31f224f810a54782055eba1e3bc`

## 이슈 요약

정적 빌드의 내장 JPEG loader가 Y/Cb/Cr의 특정 non-1:1 sampling factor 파일을 decode하지 못한다. 현재 decoder를 확장하거나 stb_image로 교체하자는 제안이다.

## main 코드 조사

- static 경로는 `src/loaders/jpg/tvgJpgd.cpp`의 내장 decoder를 사용하고 shared 빌드는 가능하면 `external_jpg`의 libturbojpeg를 사용한다.
- `jpeg_decoder::init_frame()`은 Cb/Cr sampling을 반드시 1×1로 요구하며, Y는 1×1·2×1·1×2·2×2만 허용한다. 나머지는 `JPGD_UNSUPPORTED_SAMP_FACTORS`다.
- scanline 변환 함수도 이 네 YCbCr layout별로 고정되어 있다.
- 저장소에는 stb_image가 현재 포함되어 있지 않다.

```text
SOF component sampling
Y=(1x1|2x1|1x2|2x2), Cb=1x1, Cr=1x1 -> 현재 허용
그 밖의 조합                              -> JPGD_UNSUPPORTED_SAMP_FACTORS
                                               ↓
                      MCU block order + chroma 좌표 일반화가 필요
```

## 원인 가설

**확인된 코드 제한:** decoder가 지원 sampling 조합을 명시적으로 제한한다. 제한을 푸는 것만으로는 부족하고 MCU block order와 chroma upsampling을 일반화해야 한다.

## 수정 방향과 실현 가능성

1. 첨부 JPEG의 SOF sampling factor와 progressive/baseline 여부를 fixture metadata로 고정한다.
2. 현 decoder 확장과 stb_image 대체의 `.text` 크기, decode 시간, 지원 포맷을 비교한다.
3. 확장한다면 component별 scale로 sample 좌표를 일반화하고 경계 MCU를 테스트한다.
4. libturbojpeg를 oracle로 동일 corpus를 decode한 뒤 dimensions와 RGBA를 허용 오차 내에서 비교한다.
5. replacement을 택한다면 단순 source import가 아니라 라이선스, 지원 포맷 비활성화, static `.text/.rodata`와 malformed-input 정책을 함께 review한다.

## 위험/검증

decoder는 untrusted input을 처리하므로 overflow/OOB fuzzing과 ASan이 필수다. baseline/progressive, grayscale, 기존 네 sampling, odd dimensions를 회귀 검사하고 binary-size 주장은 측정해야 한다. 초심자는 fixture와 비교 조사만 추천한다.
